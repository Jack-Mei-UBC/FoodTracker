## How to work in this repo (the loop)
This file is the source of truth. It records the invariants and gotchas that a change will otherwise get wrong ‚Äî read it before editing, and write anything non-obvious you learn back into it. The development loop is: **plan against these invariants ‚Üí implement across services ‚Üí let the `Stop` hook run `scripts/smoke-test.ps1` against the running stack ‚Üí fix any regression before finishing ‚Üí humans approve anything OCR extracted ‚Üí close the loop by updating the docs.** There is no unit-test suite; the smoke tests are the regression net, so keep them green and extend them when you add a contract. See [README.md](README.md) for the architecture and a fuller description of this loop.

**Closing the loop ‚Äî the last step of any change is documentation.** Before you report a change as done, update the docs to match reality: **CLAUDE.md** for a new/changed contract, invariant, endpoint, gotcha, schema shape, or architectural rule; **README.md** when the change alters the architecture, service roles, the data model, the page list, or the verification story. If a change genuinely touches none of those (a pure bug fix, a refactor with no contract change, a docs-only edit), say so explicitly in your final message ‚Äî "no doc update needed because ‚Ä¶" ‚Äî so the omission is a decision, not an oversight. When in doubt, add the line to CLAUDE.md; a stale spec costs the next session more than an extra sentence costs you.

**When the user corrects an approach mid-task** (wrong assumption, rejected pattern, a gotcha that bit them), add it to this file in the moment ‚Äî under Critical gotchas if it's a durable invariant, or inline near the relevant section otherwise ‚Äî rather than only fixing the immediate diff. This file is shared by every future session in this repo, so a correction recorded here won't need to be repeated.

## Project context
FoodTracker is a grocery price-intelligence app. Core entities: **stores, foods, price_logs**. The OCR service is a data-ingestion supplement ‚Äî it turns photos of receipts / shelf price tags into structured price data that maps onto the `foods` / `price_logs` schema. Nothing extracted by OCR is ever saved without passing through a human review step first.

## Services & how to run
Six containers orchestrated by `docker-compose.yml` (no root package.json ‚Äî each service is its own package):

| Service | Stack | Port | Role |
|---|---|---|---|
| `frontend` | Next.js 14 App Router, TS, Tailwind | 3000 | PWA UI |
| `backend` | Express, TS | 4000 | REST API; owns Postgres; enqueues jobs |
| `worker` | BullMQ, Node/TS | ‚Äî | Processes `scraping-queue` **and** `ocr-queue` |
| `ocr-service` | FastAPI, Python 3.12 | 8000 (loopback only) | Vision-LLM extraction via OpenRouter |
| `db` | Postgres 15 | 5432 | Relational data |
| `redis` | Redis 7 | 6379 | BullMQ queues |

```bash
docker compose up -d --build            # whole stack
docker compose up -d --build backend    # rebuild one service (do this after editing it)
docker compose logs -f worker           # follow a service's logs
docker compose exec -T db psql -U postgres -d foodtracker -c "\dt"
```

Per-service dev (outside Docker): `npm run dev | build | start` in `backend/`, `worker/`, `frontend/`. For ocr-service: `uvicorn app.main:app --port 8000` (needs `OPENROUTER_API_KEY`).

There is **no automated test suite.** Verify changes with `curl` against `http://127.0.0.1:4000`, the UI at `http://localhost:3000`, and `docker compose logs`.

## Critical gotchas
- **`db/schema.sql` only runs on a *fresh* Postgres volume** (via `docker-entrypoint-initdb.d`). On an existing DB it does **not** re-run. So every schema change must be (1) added to `schema.sql` as idempotent `ADD COLUMN IF NOT EXISTS` / `CREATE TABLE IF NOT EXISTS` / `ALTER`, **and** (2) applied to the running DB manually via `docker compose exec -T db psql -U postgres -d foodtracker <<'SQL' ... SQL`.
- **Three contracts are duplicated across languages/packages and must be kept in sync by hand** (each file says so in a comment):
  - OCR response shape: `ocr-service/app/models.py` ‚Üî `frontend/src/types/scan.ts`
  - Unit-conversion tables: `backend/src/units.ts` ‚Üî `frontend/src/lib/units.ts`
  - Nutrition scaling: `backend/src/nutrition.ts` (source of truth at commit) ‚Üî `frontend/src/lib/nutrition.ts` (live preview)
- **Frontend tsconfig targets pre-ES2015 iteration** ‚Äî spreading a `Set`/`Map` (`[...set]`) fails the build. Use `Array.from(...)` / `.concat(...)`.
- **The OpenRouter free vision model is slow (~60‚Äì90s) and flaky** (often returns `type: "unknown"`). `ocr-service/app/openrouter.py` does one reprompt-retry and degrades to `unknown` gracefully; the worker/UI expect this. For reliable/fast runs, set `OCR_MODEL` in root `.env` to a paid model (e.g. `google/gemini-2.5-flash`).
- `ocr-service` is bound to `127.0.0.1:8000` (host curl only). Internal callers use `http://ocr-service:8000` via `OCR_SERVICE_URL` (the **backend** `POST /api/scan` proxy and the worker).
- **The frontend must stay statically exportable (for the iOS/Android Capacitor build) ‚Äî do NOT add server-side code to it.** `frontend/next.config.js` turns on `output: 'export'` when `BUILD_TARGET=static` (the `npm run build:mobile` path ‚Üí `./out`), while the default `next build` stays a normal Next.js server for the Docker deploy. A static export has **no Node server**, so the frontend may have **no Next.js API routes** (`src/app/**/route.ts`), no server actions, no `getServerSideProps`-style server rendering of live data ‚Äî every page is a client component that fetches `${API_BASE_URL}/api/...` at runtime. That's why the OCR proxy lives in the backend now, not in the frontend. `NEXT_PUBLIC_API_URL` is **baked in at build time**, so the mobile build must set it to a public HTTPS backend. Capacitor scaffolding: `frontend/capacitor.config.ts` (`webDir: 'out'`), deps + `mobile:*` scripts in `frontend/package.json`; run `npx cap add android` / `npx cap add ios` (iOS needs macOS) to generate the native projects.
- **`position: fixed` overlays must render through a React portal, not inline.** Every page root `<div>` and `<main>` carries the `.animate-slide-up` CSS class (`transform: translateY(...)`, fill-mode `forwards`), and per the CSS spec any non-`none` `transform` on an ancestor makes that ancestor a new *containing block* for `position: fixed` descendants. An inline `fixed inset-0` modal therefore centers within that ancestor's full scrollable content box instead of the viewport ‚Äî it looks like it "opens in the middle of the page" and becomes unreachable once the page is taller than the viewport. The fix (and the reason `frontend/src/components/Modal.tsx` uses `createPortal(..., document.body)`) is structural, not cosmetic ‚Äî `max-h-[90vh] overflow-y-auto` alone does not fix it. Any new fixed-position overlay must go through `Modal.tsx`, not be hand-rolled inline.

## Architecture

### OCR ingestion ‚Äî two paths, both end in human review
1. **Synchronous** (`/scanner`): browser ‚Üí Express `POST /api/scan` (in-memory multer, forwards to `ocr-service POST /scan`, persists nothing) ‚Üí `ScanResponse` ‚Üí rendered in `<ReviewItems>` for edit/approve/commit. (This proxy used to be a Next.js route in the frontend; it moved to the backend so the frontend can be built as a static bundle ‚Äî see the mobile-build gotcha below. Backend env: `OCR_SERVICE_URL`.)
2. **Background queue** ("Queue for Later" / `/inbox`): browser ‚Üí Express `POST /api/scan-jobs` (multer writes the image to the shared `uploads` volume) ‚Üí `addOcrJob` enqueues `ocr-queue` ‚Üí `worker` reads the image, calls `ocr-service`, stores the `ScanResponse` JSON on the `scan_jobs` row ‚Üí `/inbox` polls status and opens the result in the **same** `<ReviewItems>` for review.

`ocr-service` sends the image **directly to a vision model** (no Tesseract/OCR engine): one call classifies `receipt | price_tag | unknown` and extracts structured fields. Model is `OCR_MODEL` (env). Pipeline: `imaging.py` (EXIF-rotate, downscale, JPEG) ‚Üí `openrouter.py` ‚Üí validated against `models.py`.

### Review & commit are centralized
`frontend/src/components/ReviewItems.tsx` is shared by the scanner and the inbox. It:
- fuzzy-matches each extracted item to the existing catalog via `frontend/src/lib/match.ts` (`bestCatalogMatch`, a dependency-free `token_set_ratio` with abbreviation/size normalization),
- flags items as `new_product` / `price_anomaly` (>30% vs latest) / low-confidence,
- computes live price-per-unit via `frontend/src/lib/units.ts` (`formatUnitPrice`),
- commits by `POST /api/foods` (create if new) + `POST /api/foods/:id/prices`.

### Data model (`db/schema.sql`)
- `foods` (name, barcode, category, unit, `image_id` ‚Äî user-chosen dashboard icon, nullable FK ‚Üí `images`, see below), `stores`.
- `price_logs`: `price`, `amount` + `amount_unit`, normalized `unit_price` (`DECIMAL(12,5)`), `is_sale`, `source` (`scan|manual|scraper|queue`), and `deleted_at` (**soft delete**).
- `audit_log`: before/after JSONB snapshot of every price_log **and consumption_log** mutation, with `reverted_at`. Revert (`POST /api/audit-log/:id/revert`) only supports `price_log` entries.
- `scan_jobs`: background OCR jobs (`status`, `image_path`, `result` JSONB, `store_id`).

### Calorie tracking
- `food_nutrition` (1:1 with `foods`): per-serving label facts (`serving_size` + `serving_unit` in the shared unit vocabulary, `calories`, macros, and micronutrients). Upsert via `PUT /api/foods/:id/nutrition`; included in `GET /api/foods` as `nutrition`. The full nutrient column set is defined once as `NUTRIENT_FIELDS` in `backend/src/nutrition.ts` ‚Äî server.ts builds its nutrition/consumption `INSERT`/`UPDATE`/`SUM` column lists from it, so **adding a nutrient = a schema migration + one entry in that array** (and matching display metadata in `frontend/src/lib/nutrition.ts` `MACRO_META`/`MICRO_META`).
- `consumption_logs`: diary entries. All nutrients (calories + macros + micros) are a **snapshot computed at log time** (`backend/src/nutrition.ts` `scaleNutrients` ‚Äî supports the diary-only unit `serving`, deliberately not added to the synced `units.ts`); editing a food's facts never rewrites history. `food_name` is denormalized; soft delete + audited (`entity_type='consumption_log'`). `GET /api/diary` sums every nutrient column for the day's micronutrient totals.
- `user_goals`: single row (id=1) of daily calorie/macro targets. `GET /api/diary?date=YYYY-MM-DD` returns a day's entries + totals + goals. `consumed_at` is a naive timestamp ‚Äî the client owns timezone handling (send local `consumed_at` and an explicit `?date=`).
- USDA FoodData Central lookup: `GET /api/nutrition-search?q=<text or barcode>` (`backend/src/fdc.ts`) proxies FDC search (key: `FDC_API_KEY` in root `.env`) and returns per-serving candidates; the nutrition editor in the dashboard food modal prefills from a picked candidate (`source: 'usda'`) and the user confirms before saving.
- `frontend/src/components/FoodDetailModal.tsx`: reusable food modal opened from a **diary** entry and the **inbox** review. Lists a food's names, prices, and nutrition; add/remove names inline (`POST/DELETE /api/foods/:id/aliases`); view a price log's attached photo in a lightbox (`image_id` ‚Üí `GET /api/images/:id`). Price and macro editing are delegated to the two shared popups below ‚Äî it only lists and launches them. Loads via `GET /api/foods/:id`.

### Meal plans (recipes)
- `meals` (name, notes, `servings` = portions the recipe makes) + `meal_ingredients` (meal_id, food_id, `amount` + `amount_unit` ‚Äî shared unit vocab plus the diary-only `serving`). **Nothing is stored for macros/cost** ‚Äî every read (`GET /api/meals`, `GET /api/meals/:id`) computes them live via `backend/src/meals.ts` `summarizeMeal`: per-ingredient `scaleNutrients` + `ingredientCost`, meal totals, and per-serving figures (totals √∑ servings), with `nutrition_complete` / `cost_complete` flags when an ingredient lacks facts or a priced purchase.
- **Cost rule** (`ingredientCost`): the food's latest non-deleted price *with a non-null `unit_price`* (through the `food_prices` join) √ó the ingredient amount in base units. Mass‚Üîvolume mismatches convert through `foods.density` (same rule as the canonical display); `serving` converts via the food's `serving_size`/`serving_unit`; count‚Üîmass/volume is unresolvable ‚Üí cost `null` for that row. Raw price only ‚Äî `usable_pct` is deliberately **not** applied.
- Endpoints: `GET/POST /api/meals`, `GET/PUT/DELETE /api/meals/:id` (PUT with `ingredients` replaces the list wholesale), `POST /api/meals/:id/clone` ("Copy of X"), `POST /api/meals/:id/log`. Meals are **hard-deleted** (not audited; clone covers recovery). `validateIngredient` (meals.ts) 400s unknown units.
- **Logging to the diary creates ONE consumption_log entry** (not one per ingredient): per-serving nutrients √ó portions, `food_id = NULL`, `food_name` = meal name, `amount_unit = 'serving'`, `source = 'meal'`, and a nullable `consumption_logs.meal_id` for provenance (`ON DELETE SET NULL` ‚Äî diary snapshots outlive the meal). It goes through the same transaction/audit path as manual entries via the shared `insertConsumptionLog` helper in server.ts ‚Äî any new diary-writing endpoint should use that helper.
- **AI meal drafting**: `POST /api/meals/generate` (`{ food_ids, targets?, notes? }`) prompts an OpenRouter chat model (`backend/src/llm.ts` ‚Äî native fetch, `response_format: json_object`, one built-in retry because free models return empty/malformed content) with the selected foods' per-serving facts; targets are **per serving of the meal**. The response is validated (unknown food_ids/units dropped) and returned as an **unsaved draft** the UI loads into the builder ‚Äî the user reviews and saves explicitly, same review invariant as OCR. Env on `backend`: `OPENROUTER_API_KEY` + `MEAL_MODEL` (root `.env`; the account has no paid OpenRouter credits, so `MEAL_MODEL` must be a `:free` model ‚Äî currently `nvidia/nemotron-nano-12b-v2-vl:free`).
- UI: `frontend/src/app/meals/page.tsx` (nav link in `layout.tsx`). The builder's live totals/cost preview mirrors the backend rules client-side (display-only; `previewCost` in the page must stay consistent with `backend/src/meals.ts` `ingredientCost`); the server's figures are authoritative. Future meal-feature ideas live in [ROADMAP.md](ROADMAP.md).

### Foods ‚Üî prices/nutrition are many-to-many
- Join tables `food_prices(food_id, price_log_id)` and `food_macros(food_id, nutrition_id)` are the authoritative food‚Üíx links for reads; `price_logs.food_id` / `food_nutrition.food_id` remain as the **origin owner** (audit + back-compat) and are kept in sync on create. A food's list/detail prices and its effective nutrition (owned preferred, else first linked) are read **through the joins**. Backfilled from the origin columns; the audit-log join still uses `pl.food_id`.
- Link/unlink an existing price or nutrition profile to additional foods: `POST/DELETE /api/foods/:id/prices/:priceLogId/link` and `POST/DELETE /api/foods/:id/macros/:nutritionId/link`.
- `food_nutrition.food_id` keeps a UNIQUE (one owned profile per food); sharing across foods happens via `food_macros`.

### Shared `Modal` component ‚Äî every popup overlay reuses this, don't fork
`frontend/src/components/Modal.tsx` is the single overlay/panel wrapper for popup modals (viewport-centered via `createPortal` into `document.body` ‚Äî see the fixed-position gotcha above; panel capped at `max-h-[90vh] overflow-y-auto` so tall content scrolls internally instead of pushing the header off-screen; Escape closes only the topmost modal via a small in-module stack for stacked modals like FoodDetailModal‚ÜíPriceEditor). Props: `{ onClose, children, maxWidth?, zClass?, backdropClass?, panelClassName? }` ‚Äî pass Tailwind classes as full literal strings (JIT requirement). Used by `PriceEditor`, `MacroEditor`, `FoodDetailModal`, `FoodIconPicker`, the dashboard price-history modal, and the meals log-to-diary popup. Image lightboxes (click-anywhere-closes, self-constraining `<img>`) are the one exception and stay hand-rolled. Any new popup must use `Modal`, not a hand-rolled `fixed inset-0` div.

### Shared class vocabulary (`globals.css` `@layer components`) ‚Äî reuse, don't re-paste
The recurring Tailwind strings are named once in `frontend/src/app/globals.css` inside `@layer components`: **`.card`** (the glass surface = `glass-panel` look + `rounded-2xl`; padding/accent/other-radius stay inline), **`.panel`** (the semi-solid `bg-slate-950/60 border border-white/5 rounded-xl` *inner* surface nested inside cards ‚Äî efficiency rows, diary tiles, meal ingredient rows), **`.field-label`**, **`.field-input`** (text input / `<select>`), **`.btn`** + **`.btn-primary`** (violet‚Üíindigo gradient) / **`.btn-secondary`** (white/5 ghost), and **`.badge`** (the small pill/tag shape ‚Äî `font-bold uppercase px-2 py-0.5 rounded-full border` ‚Äî used for category chips, status/source tags, and approval states across the dashboard, `FoodDetailModal`, `ReviewItems`, inbox, scrapes, diary, and history pages). Reuse these instead of re-pasting the raw utilities; keep the parts that genuinely vary per site (padding, width, accent color, a different radius, text size) as inline utilities ‚Äî e.g. `.badge` never bakes in color or `text-[9px]`/`text-[10px]` sizing, and a non-uppercase badge (diary's per-meal kcal chip) adds `normal-case` inline to override it.
- **They must live in `@layer components`, not as plain rules.** The components layer is emitted *before* Tailwind's utilities layer, so an inline utility still overrides the same property in one of these classes ‚Äî that's why `card rounded-3xl` renders at 24px (the hero) and `field-input focus:border-emerald-500` re-tints the focus ring. A plain (unlayered) rule defined after `@tailwind utilities` (like the standalone `.glass-panel`/`.glass-panel-hover`, which stay as-is for the hover-lift cards) would instead *win* over inline utilities ‚Äî that's exactly why the old `glass-panel border border-white/5` rendered a 0.06 white border, not the `border-white/5` you'd expect, and why `.card` bakes the 0.06 border in as raw CSS to stay pixel-identical.
- `@apply` can only reference Tailwind utilities, **not** the plain `.glass-panel` class ‚Äî so `.card` inlines the glass background/blur as raw CSS rather than `@apply glass-panel`.
- Only genuinely-repeated strings are named; one-off/compact controls (the dashboard store & add-food forms, emerald/teal accent buttons) stay inline on purpose ‚Äî don't mint a class per variant.
- `.btn` bundles a uniform `active:scale-[0.98]`/`disabled:opacity-50`; buttons converted to it gain that press/disabled feel (visually identical at rest ‚Äî an intentional consistency win, not a regression).

### Section-comment labels
Major JSX regions carry a scannable banner comment ‚Äî `{/* ‚ïê‚ïê‚ïê Section: <name> ‚ïê‚ïê‚ïê */}` ‚Äî so a region can be named directly ("edit the Summary Cards section"). Fine-grained sub-comments stay as plain `{/* ... */}`.

### Two shared popups for entering price / macros ‚Äî reuse these, don't fork
There are exactly **two** input surfaces for a food's price and nutrition, both **popup modals** (via the shared `Modal`), reused everywhere:
- `frontend/src/components/PriceEditor.tsx` ‚Äî one price log. **Add** (no `log` ‚Üí `POST /api/foods/:foodId/prices`) or **edit** (`log` ‚Üí audited `PUT /api/price-logs/:id`, optional `DELETE`). Props: `{ foodId, foodName?, log?, stores?, onClose, onSaved, onDeleted? }`.
- `frontend/src/components/MacroEditor.tsx` ‚Äî one food's per-serving nutrition facts (serving, calories, macros, collapsible micros) + USDA search. `PUT /api/foods/:id/nutrition`. Props: `{ foodId, foodName?, barcode?, nutrition?, onClose, onSaved }`.

Both are launched (never re-implemented) from: the **dashboard** food modal, the **diary**/**inbox** `FoodDetailModal`, and ‚Äî PriceEditor also ‚Äî the **history** table (Edit ‚Üí popup). If you need a price/macro form anywhere, open one of these. (`ReviewItems`' bulk scan grid is a separate concern: it edits *extracted* items pre-commit, not existing price_logs/foods.) Known names are add/rename/delete-able from the dashboard food modal and add/delete-able from `FoodDetailModal`.

### Price-per-unit normalization
`amount` + `amount_unit` (e.g. `2 lb`) normalize to a base unit (grams / ml / each) to produce `unit_price` (price per base unit) for cross-package comparison. The backend (`units.ts`, used in `POST/PUT /api/foods/:id/prices`) is the source of truth; the frontend copy is display-only.

**Canonical dashboard display (kg standard).** Every price shown on the dashboard cards / PriceEditor preview / FoodDetailModal price lines goes through `formatCanonicalUnitPrice` (`frontend/src/lib/units.ts`, display-only, no backend twin) which reads the price in **one standard unit per dimension** so rows are comparable regardless of how they were entered: **mass ‚Üí per kg**, **volume ‚Üí per kg** (via `foods.density`, see below), **count ‚Üí per each**. When the as-entered unit isn't already the canonical one it's appended in brackets **as purchased** ‚Äî total price over the purchased amount, not reduced to a per-single-unit figure (so a 600 g pack reads `$7.48/kg ($4.49/600g)`, never `$0.01/g`). The purchased amount is omitted when it's 1: `$4.41/kg ($2.00/lb)`, `$1.20/kg ($1.20/L)`, `$0.25/each ($3.00/dozen)`. This required the `GET /api/foods` `latest_prices` subquery to also select `pl.amount, pl.amount_unit` (the card computes the per-kg value client-side from price+amount+unit, not from `unit_price`). Count can't be expressed per kg without a per-item weight (not modelled), so count stays per-each.

### Density (`foods.density`)
`foods.density` (`DECIMAL(8,4)`, default 1, must be > 0) is kilograms per litre, used **only** to convert a per-volume price into a per-kg price for the canonical display above (water ‚âà 1, oil ‚âà 0.92, honey ‚âà 1.42). Display/value-only: it does **not** touch the stored `unit_price` (still per ml) nor nutrition. Accepted on `POST`/`PUT /api/foods` (validated > 0), returned via `SELECT f.*`. Edited from the dashboard food modal and `FoodDetailModal`, where the density input is shown **only when the food's `unit` is a volume unit** (`normalizeUnit(unit)?.dimension === 'volume'`); passed as `density` into `PriceEditor` for its live preview.

### Usable portion (`foods.usable_pct`)
`foods.usable_pct` (`DECIMAL(6,2)`, default 100, must be > 0) is the percent of a purchased amount that is actually usable ‚Äî e.g. 70 for chicken drumsticks (30% bone), or **>100** for dry goods that expand when prepared (dry lentils ‚Üí cooked). Edited via `PUT /api/foods/:id` (accepted on `POST` too); returned on every `GET /api/foods*` via `SELECT f.*`. It is **display/value-only**: `frontend/src/lib/units.ts` `formatUnitPrice(price, amount, unit, usablePct?)` divides the per-unit price by `usablePct/100` and suffixes " usable" (an effective cost per *usable* unit). It deliberately does **not** touch nutrition scaling (labels are already per edible serving) nor the stored `unit_price` (raw, audited, and shareable across foods via the M:N join). The efficiency endpoint compares prices within one food, so `usable_pct` cancels there ‚Äî no change. Editable from the dashboard food modal and `FoodDetailModal`; passed as `usablePct` into `PriceEditor` for its live preview.

### Dashboard food list: server-side pagination + icons
`GET /api/foods` supports optional `limit`/`offset` (plus the existing `category`/`search`, `limit` 1-100, default page size on the dashboard is 24): **omitted ‚Üí today's plain array** (required by callers that need the full catalog: `ReviewItems` fuzzy-match, the meals/diary food pickers, smoke tests ‚Äî do not add `limit` to those); **present ‚Üí `{ foods, total, categories }`**, where `total` is the filtered count and `categories` is `SELECT DISTINCT category FROM foods` (drives the dashboard's category chips so they reflect what's actually in the DB, e.g. `Scraped`/`Costco`, instead of a hard-coded list). The dashboard (`frontend/src/app/page.tsx`) debounces search 300ms and resets to page 1 on any filter change.

Every food row (both list and `GET /api/foods/:id`) also carries `display_image_id = COALESCE(foods.image_id, earliest non-deleted image_id from the food's linked price logs)` ‚Äî the dashboard card thumbnail. `foods.image_id` (nullable FK ‚Üí `images`) is the user's explicit icon choice, set via the popup opened by clicking a card's thumbnail (`frontend/src/components/FoodIconPicker.tsx`): pick one of the food's saved scan/scrape images (fetched via the existing `GET /api/foods/:id/prices`, deduped by `image_id`) or upload a new one, then crop to a square with `react-easy-crop` (client-side canvas crop, ‚â§512√ó512 JPEG) before `POST /api/images` + `PUT /api/foods/:id { image_id }`. `PUT /api/foods/:id` is otherwise pure `COALESCE` (omitted field = unchanged), but `image_id` is **presence-based** since COALESCE can never express "set to null": key absent ‚Üí untouched, `{image_id: null}` ‚Üí clears back to the `display_image_id` fallback (the picker's "Reset to default"), `{image_id: <n>}` ‚Üí set (400 if it doesn't reference a saved `images` row). Re-cropping never deletes the previous `images` row (images are shared/immutable elsewhere too ‚Äî scrapes, scans); orphaned icon images are an accepted tradeoff, not a leak to fix.

### History, audit & revert
Every price_log create/update/delete flows through `backend/src/audit.ts` `recordAudit` **inside the same transaction** as the mutation. Deletes are soft (`deleted_at`); revert (`POST /api/audit-log/:id/revert`) applies the inverse and is itself audited (so reverts are revertible). Any query that surfaces a "current" price ‚Äî `GET /api/foods` `latest_prices` subquery and `GET /api/prices/efficiency` ‚Äî must filter `deleted_at IS NULL`. UI: `frontend/src/app/history/page.tsx`.

### Scraper (Flipp flyers)
`worker/src/worker.ts` also processes `scraping-queue` (triggered by `POST /api/scrape/:storeId`, body `{ postal_code?, query? }`; postal code falls back to `FLIPP_POSTAL_CODE` in root `.env`). `worker/src/flipp.ts` hits Flipp's public flyer JSON API (`backflipp.wishabi.com/flipp/items/search` ‚Äî the flipp.com backend; **no Playwright/browser**, the worker image is plain `node:20-slim`) and keeps items whose merchant fuzzy-matches the store name ‚Äî so the store must be named like the Flipp merchant ("Walmart", "Save-On-Foods"). With a `query` it logs up to 15 matching deals (creating foods with category `Scraped` when nothing in the catalog matches); without one it searches each catalog food (capped at 50, rate-limited) and logs the single best-scoring current deal per food. Pack sizes are parsed from flyer item names/affixes ("473mL", "2 x 96g", "/lb", "2/" multi-buys) into `amount` + `amount_unit`. The worker writes **through the backend REST API** (`BACKEND_URL`, i.e. `POST /api/foods/:id/prices` with `source: 'scraper'`) ‚Äî never direct SQL ‚Äî so scraped prices get unit normalization, their `food_prices` join row (without it they'd be invisible to reads), and an audit entry like every other source.

**Progress tracking + source images.** `POST /api/scrape/:storeId` first inserts a `scrape_jobs` row (`db/schema.sql`) and passes its id on the queue job as `scrapeJobId`; the response includes `scrapeJobId`. The worker updates that row's `status`/`phase`/`total`/`processed`/`logged` as it runs and **appends one detail record per logged price** to `items` JSONB (food, flyer name, price, `image_id`, `flyer_url`). These progress writes go **direct via the worker's pg pool** (bookkeeping, like the OCR path's `scan_jobs` updates) ‚Äî only the *prices* go through the audited backend API. For each logged price the worker downloads the item's Flipp **clipping image** (`clipping_image_url`) and registers it as an `images` row by POSTing multipart to `POST /api/images` (reusing the photo path), then attaches the returned `image_id` to the price log ‚Äî so scraped prices show their real flyer artwork in the existing `image_id` lightbox everywhere (FoodDetailModal, history). `flipp.ts` `flippItemUrl` builds `https://flipp.com/en-ca/item/<flyer_item_id>` ‚Äî the deep link to the flyer page the item came from (**Flipp's JSON API exposes only per-item clipping images, no full-page rasters**, so "the relevant page" is a link, not a saved image). Reads: `GET /api/scrape-jobs` (list, `item_count` only) and `GET /api/scrape-jobs/:id` (full `items`). UI: `frontend/src/app/scrapes/page.tsx` ‚Äî live progress list (polls every 2s while any job is `queued`/`processing`), expandable per-run to show each logged price with its saved flyer thumbnail (lightbox) and a "Flyer page ‚Üó" link; the dashboard scrape form redirects to `/scrapes?job=<id>` (auto-expands, needs a `<Suspense>` boundary for `useSearchParams`).

### Second scraper source: cocowest.ca Costco sale posts
`worker/src/cocowest.ts` scrapes cocowest.ca "weekend update" blog posts, which list Costco BC/AB/SK/MB flyer sale items. Triggered by `POST /api/scrape-cocowest`, body `{ store_id, url }` (`url` must match `cocowest.ca`) ‚Äî no postal code or merchant matching, since the whole post belongs to whichever store the caller selects (create a "Costco" store first). Shares the same `scrape_jobs` table and `scraping-queue` as Flipp: `backend/src/queue.ts` `addCocowestScrapeJob` enqueues `{ ..., source: 'cocowest', url }`, and `worker/src/worker.ts`'s queue handler branches on `job.data.source` (`'flipp'` is the default for back-compat with jobs already in flight) to call `scrapeCocowest` instead of `scrapeFlippFlyers`. `scrapeCocowest` mirrors Flipp's *query mode*: log a price for **every** parseable item, creating a food with **category `Costco`** when nothing in the catalog fuzzy-matches (the confirmed scope choice ‚Äî the post mixes grocery with electronics/appliances/toys, and everything gets logged rather than filtered).

- **No JSON API, no DOM parser.** cocowest is a WordPress blog; each item is one `<img>` whose `alt` text carries `ITEM# NAME SIZE ($SAVINGS INSTANT SAVINGS EXPIRES ON YYYY-MM-DD)? $PRICE` and whose `data-src` (not `src`, which is an inline SVG lazyload placeholder) is the real product photo. `cocowest.ts` parses this with regex, not an HTML/DOM parser ‚Äî it keeps the worker's plain `node:20-slim` image dependency-free, and is actually more robust here: some product names contain an unescaped `"` (e.g. `ACER 27"...`), which truncates the `alt` attribute in the raw HTML itself (not a parser bug). Those items have no trailing `$price` after truncation and are silently skipped ‚Äî in practice this is ~40 of ~380 images per post, almost all non-grocery electronics/TVs.
- **Shared parsing/matching.** Amount/unit parsing (`parseAmount`) and catalog fuzzy-matching (`nameMatchScore`, `symmetricMatchScore`) were pulled out of `flipp.ts` into `worker/src/scrape-common.ts` so both scrapers use one implementation; `flipp.ts`'s `parseFlippAmount` is now a thin wrapper that reads the multi-buy quantity from Flipp's `pre_price_text` and delegates. `scrape-common.ts` also adds a `PACK OF (\d+)` pattern (cocowest phrasing) alongside Flipp's `N PK`/`N CT` forms.
- **Images and progress.** Per logged item, `saveFlyerImage(item.image_url)` downloads the cocowest product photo and registers it via `POST /api/images` exactly like the Flipp path, so cocowest-sourced prices get the same `image_id` lightbox everywhere. `flyer_url` in each `scrape_jobs.items` record is the cocowest **post URL** (not a per-item deep link ‚Äî cocowest has no per-item page).
- **Schema:** `scrape_jobs.source` (`'flipp'|'cocowest'`, default `'flipp'`) and `scrape_jobs.source_url` (cocowest's post URL) ‚Äî added via the usual `ADD COLUMN IF NOT EXISTS` pattern in `db/schema.sql`, so existing DBs need the manual `ALTER TABLE` too (see the schema.sql gotcha above).
