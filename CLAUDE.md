# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## How to work in this repo (the loop)
This file is the source of truth. It records the invariants and gotchas that a change will otherwise get wrong — read it before editing, and write anything non-obvious you learn back into it. The development loop is: **plan against these invariants → implement across services → let the `Stop` hooks typecheck edited services and run `scripts/smoke-test.ps1` against the running stack → fix any regression before finishing → humans approve anything OCR extracted → close the loop by updating the docs.** There is no unit-test suite; the smoke tests are the regression net, so keep them green and extend them when you add a contract. See [README.md](README.md) for the architecture and a fuller description of this loop.

**The loop is wired into tooling, not just described here** (`.claude/settings.json` + `scripts/hooks/`):
- **PreToolUse guard** (`scripts/hooks/pre-bash-guard.ps1`): denies state-destroying commands — `docker compose down -v` / `docker volume rm|prune` (the postgres volume IS the price history), force-pushes, and `DROP`/`TRUNCATE` via psql. If a denied command is genuinely needed, the *user* runs it.
- **Doc-sync push gate** (same script): `git push` of a branch that changes **>5 files or >150 lines** vs `origin/main` without touching any doc file is denied — run the **doc-sync agent** (below) first, or bypass deliberately with `SKIP_DOC_GATE=1 git push ...` when you've stated in your final message why no doc update is needed.
- **Stop hooks**: `scripts/hooks/stop-typecheck.ps1` (per-service `tsc --noEmit`, only for services with uncommitted changes) then `scripts/smoke-test.ps1`. Both use a 20s marker debounce; exit 2 feeds the failure back — no green, no done.

**Four subagents live in `.claude/agents/` — they ARE steps of the loop, not optional extras.** Launch them at these checkpoints:
| Agent | When |
|---|---|
| **contract-guard** (read-only) | After touching ANY side of a hand-synced contract pair (see Critical gotchas); before merging a big PR. |
| **invariant-reviewer** (read-only) | Before finishing any multi-file change — reviews the diff against this file's invariants. |
| **doc-sync** (edits docs only) | Before pushing/PRing anything beyond a trivial fix; mandatory when the push gate fires. |
| **verifier** | Before opening a PR / after a cross-service change — runs the full ladder (tsc ×3, STRICT smoke, Playwright when frontend changed, `build:mobile` static gate). |

**Closing the loop — the last step of any change is documentation.** Before you report a change as done, update the docs to match reality: **CLAUDE.md** for a new/changed contract, invariant, endpoint, gotcha, schema shape, or architectural rule; **README.md** when the change alters the architecture, service roles, the data model, the page list, or the verification story. If a change genuinely touches none of those (a pure bug fix, a refactor with no contract change, a docs-only edit), say so explicitly in your final message — "no doc update needed because …" — so the omission is a decision, not an oversight. When in doubt, add the line to CLAUDE.md; a stale spec costs the next session more than an extra sentence costs you.

**When the user corrects an approach mid-task** (wrong assumption, rejected pattern, a gotcha that bit them), add it to this file in the moment — under Critical gotchas if it's a durable invariant, or inline near the relevant section otherwise — rather than only fixing the immediate diff. This file is shared by every future session in this repo, so a correction recorded here won't need to be repeated.

## Project context
FoodTracker is a grocery price-intelligence app. Core entities: **stores, foods, price_logs**. The OCR service is a data-ingestion supplement — it turns photos of receipts / shelf price tags into structured price data that maps onto the `foods` / `price_logs` schema. Nothing extracted by OCR is ever saved without passing through a human review step first.

## Services & how to run
Six containers orchestrated by `docker-compose.yml` (no root package.json — each service is its own package):

| Service | Stack | Port | Role |
|---|---|---|---|
| `frontend` | Next.js 14 App Router, TS, Tailwind | 3000 | PWA UI |
| `backend` | Express, TS | 4000 | REST API; owns Postgres; enqueues jobs |
| `worker` | BullMQ, Node/TS | — | Processes `scraping-queue` (concurrency 1) **and** `ocr-queue` (parallel, multi-model pool) |
| `ocr-service` | FastAPI, Python 3.12 | 8000 (loopback only) | Vision-LLM extraction via OpenRouter |
| `db` | Postgres 15 | 5432 | Relational data |
| `redis` | Redis 7 | 6379 | BullMQ queues |

```bash
docker compose up -d                    # DEV stack (hot reload) — see below
docker compose up -d --build backend    # rebuild one service (needed after a package.json change)
docker compose logs -f worker           # follow a service's logs
docker compose exec -T db psql -U postgres -d foodtracker -c "\dt"
node frontend/e2e/fixtures/seed.mjs              # seed the fixture catalog (see below)
```

**Compose is split three ways, and the split is load-bearing:**
- **`docker-compose.yml`** — base, production-shaped. Each service runs its image's own `CMD` (Dockerfile `prod` target), no bind mounts.
- **`docker-compose.override.yml`** — the DEV overlay, **loaded automatically** by a bare `docker compose up`. Switches to the Dockerfile `dev` target, bind-mounts source, and runs the watcher. This is what gives you hot reload; your day-to-day command doesn't change.
- **`docker-compose.ci.yml`** — used as `docker compose -f docker-compose.yml -f docker-compose.ci.yml up -d --wait …`. Naming both files **suppresses the automatic dev override**, so CI tests the real deploy artifact.

**After a dependency change, `--build` alone is NOT enough — you also need `-V`.** The dev override masks `node_modules` with an *anonymous volume*, and that volume **survives a rebuild**, so the new image's modules stay hidden behind the stale ones. The symptom is a `Cannot find module 'x'` for a package that is demonstrably in `package.json`. The correct command is `docker compose up -d --build -V <svc>` (`-V` = `--renew-anon-volumes`).

**Dependencies are installed in the image, never in a container start command.** The old `command: sh -c "npm install && npm run dev"` re-installed on every boot and discarded the image layer, which is why cold starts took minutes (they now take ~25s with `--wait`). Hot reload still works because the source bind mount is paired with an **anonymous volume on `/usr/src/app/node_modules`**, which masks the host directory and preserves the image's modules. **The tradeoff: a `package.json` change now needs `docker compose up -d --build <svc>`, not just a restart.**

All healthchecks are real (`pg_isready`, `redis-cli ping`, `/api/health`), so `docker compose up -d --wait` blocks until the stack is genuinely ready — no poll loops needed.

Per-service dev (outside Docker): `npm run dev | build | start` in `backend/`, `worker/`, `frontend/`. For ocr-service: `uvicorn app.main:app --port 8000` (needs `OPENROUTER_API_KEY`).

There is **no unit-test suite** (a Vitest secondary for the pure-logic libs is planned — see [SHADCN-MIGRATION.md](SHADCN-MIGRATION.md)). Verify changes with `curl` against `http://127.0.0.1:4000`, the UI at `http://localhost:3000`, and `docker compose logs`. `scripts/smoke-test.ps1` is the read-only regression net (Stop hook); `scripts/smoke-test.sh` is its hand-synced twin that CI runs. **Both support `STRICT=1`** — fail (exit 2) instead of skip when the stack is down; CI sets it, and the non-strict skip prints an unmissable "NOTHING WAS VERIFIED" banner so a green turn with no stack is never mistaken for a verified one.

**There IS now a UI test net: Playwright, in `frontend/e2e/`** (see its [README](frontend/e2e/README.md)). Run it with the stack up: `docker compose up -d --wait`, then `npm run seed && npm run test:e2e` from `frontend/`. It's **not** wired to the Stop hook (too slow) — the hook keeps running the PowerShell smoke test. Three layers: route smoke (every page loads, no console errors, every page-root `data-loc` present), **interaction contracts** (modal centering/stacking/backdrop, dashboard search/sort/filter, the per-100 kcal basis, the expired-sale rule), and — later — visual snapshots. **The interaction layer is style-agnostic on purpose**: it must pass unchanged across a restyle, so a test there needing edits means behavior broke, not that the test is stale. It lives in `frontend/` because there's no root package.json, so a spec at the repo root couldn't resolve `@playwright/test`.

**Fixtures live in `frontend/e2e/fixtures/`.** `seed-data.mjs` is the single static source of truth for the dev/CI catalog; `seed.mjs` inserts it **through the REST API** (raw SQL would skip `unit_price` normalization and the `food_prices` join rows, producing fixtures that look right in the DB and wrong in the UI). It's idempotent — matched by barcode/name, so re-running is a no-op. The set deliberately covers every display branch: mass/volume-with-density/count units, a food with no nutrition, `usable_pct != 100`, an alias, an archived food, and **an active sale plus an expired one** so the `ACTIVE_PRICE_SQL` invariant is actually exercised. Tests import the same file, so expectations can't drift from the data.

**An empty catalog is not a regression** — it's an unseeded stack. Both smoke scripts `[SKIP]` their catalog-dependent assertions in that case and tell you to run the seeder, rather than failing. They also **discover the food id** from `GET /api/foods` instead of hard-coding `1`/`5`, which don't survive a fresh volume. To exercise the **OpenRouter-backed AI surfaces** by hand — raw key/credits check, OCR vision (`ocr-service /scan`), meal drafter (text pool), audit auto-tagger (`TAG_MODEL`|pool), duplicate-finder (`MERGE_MODEL`|pool) — run `scripts/manual-ai-tests.ps1` (`-Test openrouter|ocr|meal|tag|merge|all`, `-Image <receipt.jpg>` for the OCR path). It makes **real, token-costing, slow (~60-90s on `:free`) calls**, so it's deliberately *not* wired to any hook — run it manually.

## Critical gotchas
- **`db/schema.sql` only runs on a *fresh* Postgres volume** (via `docker-entrypoint-initdb.d`). On an existing DB it does **not** re-run. So every schema change must be (1) added to `schema.sql` as idempotent `ADD COLUMN IF NOT EXISTS` / `CREATE TABLE IF NOT EXISTS` / `ALTER`, **and** (2) applied to the running DB manually via `docker compose exec -T db psql -U postgres -d foodtracker <<'SQL' ... SQL`.
- **Three contracts are duplicated across languages/packages and must be kept in sync by hand** (each file says so in a comment):
  - OCR response shape: `ocr-service/app/models.py` ↔ `frontend/src/types/scan.ts`. **`price_tag` data is `{ store_name, items: PriceTagItem[] }`** — a shelf photo routinely shows several tags, so it is a LIST, not one flat tag. Rows written before that change still hold the old flat object in `scan_jobs.result`, so **every reader must tolerate both**: `PriceTagData` has a pydantic `model_validator(mode="before")` that wraps a flat tag into `items[]`, `lib/scanResult.ts` falls back to the flat shape, and the two **item-count** sites do too — `worker/src/worker.ts` `resultItemCount` (its quality gate rejects a scan with 0 items, so getting this wrong makes the worker retry every valid multi-tag scan to death) and the `GET /api/scan-jobs` `item_count` CASE.
  - Unit-conversion tables: `backend/src/units.ts` ↔ `frontend/src/lib/units.ts`
  - Nutrition scaling: `backend/src/nutrition.ts` (source of truth at commit) ↔ `frontend/src/lib/nutrition.ts` (live preview)
  - **Smoke tests: `scripts/smoke-test.ps1` (the Stop hook) ↔ `scripts/smoke-test.sh` (the CI gate).** These drifted badly once — the bash copy had fallen ~24 assertions and 4 frontend routes behind, so **"CI is green" stopped meaning "the hook would pass."** Add an assertion to both or neither.
- **Frontend tsconfig targets pre-ES2015 iteration** — spreading a `Set`/`Map` (`[...set]`) fails the build. Use `Array.from(...)` / `.concat(...)`.
- **An empty `FREE_IMAGE_MODELS` silently makes OCR sequential.** The worker's ocr-queue concurrency *is* `poolSize()` = the number of distinct image models, and an empty list seeds from the legacy singular `OCR_MODEL` → **1 model → 1 lane → one receipt at a time**. The parallel machinery can be fully built and still behave sequentially purely from config, so when scanning feels serial, check the env before the code (`docker compose exec worker printenv | grep MODELS`). Same shape on the text side: an empty `FREE_TEXT_MODELS` means a single model with **no failover**, so one stalled free model hangs the whole request. Both lists are populated in `.env`/`.env.example` with models verified against a real receipt / a real JSON tagging prompt — **re-verify before adding a model**: the OpenRouter catalog tags `google/lyria-3-*` (music) and `nvidia/nemotron-3.5-content-safety` as image-input even though they're useless as VLMs, and `google/gemma-4-31b-it:free` 429s on this account (a lane that always fails just burns a scan's retry budget).
- **`backend/src/llm.ts` is a thin shim over `chatJsonPool`** — it must **re-export `LlmError` from `modelPool.ts`, not define its own**. `server.ts` does `err instanceof LlmError` to pass the LLM status through; a second copy of the class makes that check silently false and turns every LLM failure into a generic 500. Routes should report the model from `chatJson`'s `onModelUsed` callback (the model that actually answered), **not** `process.env.*_MODEL`, which says nothing about which model in the failover chain won.
- **Model pool — the 4-list env contract, duplicated across the worker (image) and backend (text) by hand.** Model selection is centralized in two twin modules that both parse the same comma-separated env lists — `FREE_IMAGE_MODELS`/`PAID_IMAGE_MODELS` (`worker/src/modelPool.ts`) and `FREE_TEXT_MODELS`/`PAID_TEXT_MODELS` (`backend/src/modelPool.ts`), plus shared `USE_PAID_MODELS`. An empty `FREE_*` list **seeds from the legacy singular env** (`OCR_MODEL` / `MEAL_MODEL`) so old `.env` files keep working. Add/remove models by editing env only — no caller ever hard-codes a model. See the OCR-ingestion and meal-drafting sections for how each pool runs (image = parallel across scans; text = sequential failover). **The two `parseList`/seed helpers are hand-synced** — keep them in step like the other cross-package contracts.
- **Never `json.loads()` a model reply whole — models don't stop cleanly.** `ocr-service/app/openrouter.py` `_extract_json_object` exists because a real receipt scan was degraded to `type: unknown` while its `raw_text` held a complete, correct receipt. Three failure shapes, all seen in practice: trailing chatter after the object (`json.loads` → "Extra data"), the reply cut off at `max_tokens` (→ "Unterminated string"), and — the nasty one — the model **looping**, re-emitting `type`/`confidence`/`data` several times inside the *same* object until it ran out of tokens. That last one means **`object_pairs_hook` must keep the FIRST duplicate key**, since Python's last-wins default would hand back the truncated final repeat and throw away the good answer the model gave first. So the parser: strips fences → seeks the first `{` → `raw_decode` (one value, ignore the rest) → falls back to `_repair_truncated_json`, which closes the containers left open at the last clean value boundary. Everything still goes to human review, so a receipt recovered a few lines short beats an empty result.
- **A field the prompt only declares in the JSON schema tends to come back null — it needs an explicit rule, and for anything fiddly a worked example.** `price_tag` listed `amount`/`amount_unit` in its shape but had no rules block (receipts had one), and every model returned `amount: null` for a Costco tag whose weight was plainly legible at the downscaled resolution the model receives (verified by rendering the 1792px image — so this was never a resolution problem). Adding a rules block helped only the strongest model; what actually fixed it across models was a **worked example** in `prompts.py` showing a real two-tag shelf photo and its expected JSON. After that, gemma-4-26b and nemotron-3-omni both returned the exact weights and per-tag unit prices. When adding such an example, **verify on a DIFFERENT image** that the model generalizes rather than parroting the example's numbers (checked here against an unrelated yogurt tag: `2.2 kg`, sale price, and EXP date all read correctly).
- **The OpenRouter free vision models are slow (~60–90s) and flaky** (often return `type: "unknown"`). The worker's image pool mitigates this by running **all `FREE_IMAGE_MODELS` in parallel across different scans** and **retrying a scan on the next model** when one fails (up to `OCR_MAX_ATTEMPTS`); `ocr-service/app/openrouter.py` still does one reprompt-retry per call and degrades to `unknown` with `raw_text` set. For reliable/fast runs, add a paid model to `PAID_IMAGE_MODELS` and enable it (per-batch toggle on `/staging`, or `USE_PAID_MODELS=true`).
- `ocr-service` is bound to `127.0.0.1:8000` (host curl only). Internal callers use `http://ocr-service:8000` via `OCR_SERVICE_URL` (the **worker's** background OCR calls; it passes the chosen `model` per request). `ocr-service` is a **dumb per-model executor** — the worker owns model selection.
- **The frontend must stay statically exportable (for the iOS/Android Capacitor build) — do NOT add server-side code to it.** `frontend/next.config.js` turns on `output: 'export'` when `BUILD_TARGET=static` (the `npm run build:mobile` path → `./out`), while the default `next build` stays a normal Next.js server for the Docker deploy. A static export has **no Node server**, so the frontend may have **no Next.js API routes** (`src/app/**/route.ts`), no server actions, no `getServerSideProps`-style server rendering of live data — every page is a client component that fetches `${API_BASE_URL}/api/...` at runtime. That's why all server work (OCR, everything else) lives in the backend, not in the frontend. `NEXT_PUBLIC_API_URL` is **baked in at build time**, so the mobile build must set it to a public HTTPS backend. **This is also why testing from a phone on the LAN fails with "backend server unreachable" while the page itself loads fine**: the value is inlined into the *client* bundle, so it must be an address the **browser** can reach, not one the container can. The default `http://localhost:4000` makes a phone call *itself*. Fix: set `NEXT_PUBLIC_API_URL=http://<host-LAN-IP>:4000` in `.env` and recreate the frontend (`docker compose up -d --force-recreate frontend`) — the compose override reads it through `${NEXT_PUBLIC_API_URL:-…}`. The backend already binds `0.0.0.0:4000` and runs wide-open `cors()`, so nothing else needs changing; if it still fails after this, check the host firewall for inbound 4000. (Note `ocr-service` is deliberately loopback-only and is *not* LAN-reachable.) Capacitor scaffolding: `frontend/capacitor.config.ts` (`webDir: 'out'`), deps + `mobile:*` scripts in `frontend/package.json`; run `npx cap add android` / `npx cap add ios` (iOS needs macOS) to generate the native projects.
- **`position: fixed` overlays must render through a React portal, not inline.** Every page root `<div>` and `<main>` carries the `.animate-slide-up` CSS class (`transform: translateY(...)`, fill-mode `forwards`), and per the CSS spec any non-`none` `transform` on an ancestor makes that ancestor a new *containing block* for `position: fixed` descendants. An inline `fixed inset-0` modal therefore centers within that ancestor's full scrollable content box instead of the viewport — it looks like it "opens in the middle of the page" and becomes unreachable once the page is taller than the viewport. The fix (and the reason `frontend/src/components/Modal.tsx` uses `createPortal(..., document.body)`) is structural, not cosmetic — `max-h-[90vh] overflow-y-auto` alone does not fix it. Any new fixed-position overlay must go through `Modal.tsx`, not be hand-rolled inline.

## Architecture

### OCR ingestion — one queue path, ending in human review
There is a **single** ingestion path (the old synchronous `POST /api/scan` + inline-review scanner was removed): intake → **staging** → background OCR (multi-model) → **inbox** review.
1. **Intake** (`/scanner`, `frontend/src/app/scanner/page.tsx`): a pure uploader — click/drop/paste → `POST /api/scan-jobs` writes each image to the shared `uploads` volume and inserts a **`staged`** `scan_jobs` row **without enqueuing**. It renders no results and no `<ReviewItems>`.
2. **Staging** (`/staging`, `frontend/src/app/staging/page.tsx`): lists `staged` jobs; the user optionally re-crops each (shared `<ImageCropper>` → `PUT /api/scan-jobs/:id/image`, stores the crop linked to the original and repoints the job) and **sends jobs for processing** (`POST /api/scan-jobs/process` `{ ids?, use_paid }` → `staged`→`queued`, writes `use_paid` on each row, `addOcrJob` enqueues `ocr-queue`). A **"Use paid models"** checkbox sets `use_paid` per batch.
3. **Worker (multi-model)**: `worker/src/worker.ts` processes `ocr-queue` at **concurrency = pool size**, so several scans run **in parallel, each on a different model**. Per scan, `processScanJob` cycles the job's model list (`worker/src/modelPool.ts`, round-robin start) up to `OCR_MAX_ATTEMPTS`, `POST`ing `ocr-service /scan` with an explicit `model` each time; it accepts the first **acceptable** result (`type != unknown` && ≥1 item) and otherwise keeps the best-effort one (preserving `raw_text`). The result JSON lands on the row.
4. **Inbox** (`/inbox`): polls status and opens the stored `ScanResponse` in `<ReviewItems>` for edit/approve/commit — including `unknown`/no-item results, which render the model's `raw_text` for copy-paste / manual entry (this is where scan-failure recovery lives now).
5. **Recovery from a bad scan** — two paths, both from the inbox:
   - **See what every model read.** The worker keeps only the single best body on `scan_jobs.result`, so it also writes **`scan_jobs.attempts`** — one record per model tried (`{ model, ok, type, item_count, confidence, raw_text, error }`), written on **both** the `done` and `failed` paths. `<RawModelOutput>` renders the winning `raw_text` plus every attempt. This matters more than it sounds: a scan can be marked `unknown` while a model's `raw_text` holds a perfectly good receipt (ocr-service failed to *parse* it, not to *read* it), and before this that text was the only copy and it was thrown away.
   - **Re-crop it.** **`POST /api/scan-jobs/:id/restage`** sends a processed job back to `/staging`: resets `status`→`staged`, clears `result`/`attempts`/`error`/`processed_at`, and **repoints the job at the crop's original** (`images.original_image_id`) so the re-crop starts from the full photo instead of tightening a bad crop. Returns `reverted_to_original`. 404 unknown, 400 when already `staged`/`discarded` or still `queued`/`processing` (the worker is mid-read). The superseded crop's `images` row is left orphaned — same accepted tradeoff as the icon picker. **Not audited** (no `price_log` is touched).

`ocr-service` is a **dumb per-model executor**: `POST /scan` takes the image + an explicit `model` (falls back to `OCR_MODEL` only for direct host/manual calls) and sends it **directly to that one vision model** (no Tesseract) — classify `receipt | price_tag | unknown` + extract structured fields. Pipeline: `imaging.py` (EXIF-rotate, downscale, JPEG) → `openrouter.py` → validated against `models.py`. It degrades an unparseable-after-retry response to `type: 'unknown'` **with `raw_text`** set (never a 502), so the text always reaches the inbox. Model selection + parallelism + retries all live in the worker's pool, not here.

**Crop-before-OCR (staging only).** Cropping happens on `/staging` (`modal.staging-crop`, shared `<ImageCropper>`) via `PUT /api/scan-jobs/:id/image`, which stores the crop linked to the original (`images.original_image_id`, nullable self-FK) and repoints the job. The **cropped** image is what gets OCR'd and what committed `price_logs.image_id` reference; the **full original is stored too**. **GPS is read from the original** upload, never the crop — a canvas crop strips EXIF — so store auto-match still works.

### Review & commit are centralized
`frontend/src/components/ReviewItems.tsx` is the review surface for the **inbox** (`/inbox`) — the single place OCR results are edited/approved/committed now that the synchronous scanner path is gone. It:
- matches each item to the catalog via `ReviewItems`' `computeMatch` — an **exact barcode match wins** over the fuzzy name match (`frontend/src/lib/match.ts` `bestCatalogMatch`, a dependency-free `token_set_ratio`), so an item links to its food even when the OCR'd name is unrecognizable ("SM SF STEEVE SYRUP" ↔ "Steeves Maple Syrup"). Runs on scanned items **and hand-edited names** (re-match on blur), so nothing silently creates a duplicate,
- **manual match fallback** when auto-match misses or picks the wrong food: "match…" / "change match" on every grid row (and "match to existing…" on a `new_product` flag) opens `modal.match-item` — a catalog search (name/barcode) whose pick calls `matchItemTo` → `reviewFor(...)`, setting `existingFoodId` + `manualMatch` (shown as "(chosen)", no fuzzy %). "Create as a new item" (`unmatchItem`) forces a new food instead. **`manualMatch` makes the choice sticky** — `matchNameForItem` skips re-matching so a later name edit can't clobber it. Committing a manually-matched item logs the price against the chosen food and (via the existing alias path) **learns the scanned name as an alias**, so the same line auto-matches next time. It's a `Modal`, not an inline dropdown: the grid sits in an `overflow-x-auto` wrapper, and per CSS a non-`visible` overflow-x forces overflow-y to `auto`, which would clip an absolutely-positioned list,
- flags items as `new_product` / `price_anomaly` (>30% vs latest) / low-confidence,
- **has ONE unit control per row, on the Amount cell.** The grid's separate free-text "Unit" column (the food's own `foods.unit`) was removed — it was indistinguishable from the amount's unit and only ever mattered when the row created a new food. `foods.unit` is now **derived** from the purchase unit at commit (`catalogUnitFor`: the row's `amountUnit`, with count units collapsed to `each`, falling back to the scanned `item.unit`), so a row entered as `600 g` creates a mass food instead of the scan's stale `each`. The **Amount** box stays free-text ("2lb" fills both fields) and must keep its own `amountText` draft state: driving that input from the parsed `amount` alone re-renders the number mid-typing and eats the half-finished suffix, so `lb` could never be completed (you'd get `2 L`). `amountText` is display-only, cleared on blur, never committed.
- lets you pick the **Store Bought** (auto-selected from photo GPS when within ~200m), plus inline **create** and **remove** of stores so a scan from an untracked store commits without leaving the review: "+ New" posts `POST /api/stores {name}`; "Remove" opens `modal.store-remove`, which loads `GET /api/stores/:id/usage` to show what references the store and offers a **"Move its items to"** picker (another store, or "Leave unassigned") before calling `DELETE /api/stores/:id { reassign_to }` — see the stores note in the data model. The panel then follows the reallocation target (or falls back to a remaining store) if the removed one was selected,
- **store changes are confirmed, and can be applied to every open review at once.** Changing the `<select>` doesn't apply immediately — it stages the pick and opens `modal.store-change` ("Change store?") carrying an **"Apply to all N open reviews" checkbox that defaults to checked** (shown only when >1 panel is open; the button reads "Change all" vs "Change"). Creating a store routes through the same confirm when >1 panel is open. This is the batch case: a whole staging batch scanned at one store gets its store set once. Mechanically the **inbox owns the store** — `openReviews[].storeId` is the source of truth, passed down as the controlled `storeId` prop; `ReviewItems` calls `onStoreChange(storeId, applyToAll)` and the inbox's `handleStoreChange` updates either that one review or all of them, which flows back down and re-syncs each panel's selector. GPS auto-select deliberately does **not** broadcast (it's per-photo),
- lets you **search the catalog to add** an existing food (by name/barcode), prefilled with its latest price, and computes live price-per-unit via `frontend/src/lib/units.ts` (`formatUnitPrice`),
- commits by `POST /api/foods` (create only when unmatched by barcode/name) + `POST /api/foods/:id/prices`. `foods.barcode` is UNIQUE, so a create with a clashing barcode returns **409** (not a raw 500); commit **surfaces the real per-item server error** to the toast + console and keeps failed rows in the list to retry.
- shows the source photo(s) via **`<ScanImages>`** and every model's output via **`<RawModelOutput>`** (both shared — see below). The raw-output panel is **collapsed by default on every review, including successful ones** (a misread price is worth checking against the model's own text) and **auto-expanded when nothing parsed**, which is when it's the main event. It is no longer gated on `parsedItems.length === 0`.
- offers **"Re-crop in Staging"** (the `onRestage` prop → `POST /api/scan-jobs/:id/restage`) whenever no items were detected, so a bad crop is fixable without discarding and re-uploading.

The inbox pairs this component with `frontend/src/lib/scanResult.ts` (`scanResultToRawItems` — the one ScanResponse→RawItem[] mapping) and `frontend/src/components/StatusToast.tsx` (`useToast` + `<StatusToast>`).

### Catalog audit (bulk clean-up of non-food)
The cocowest scraper deliberately logs **every** item in a Costco flyer post, so the catalog fills with non-food (phones, luggage, detergent) — `/audit` is where that gets cleaned up in bulk.
- **`foods.deleted_at` is a soft delete** (same convention as `price_logs`/`consumption_logs`/`receipts`; foods are **never** hard-deleted, so archive/restore *is* the recovery path — there is no `DELETE /api/foods/:id`). **Every catalog read must filter `f.deleted_at IS NULL`**: `GET /api/foods` (via the shared `appendFoodFilters`, which also drives the count + the `categories` list) and `GET /api/prices/efficiency`. Deliberately **not** filtered: `GET /api/foods/:id` and the meals/diary joins, so an archived food still resolves by id and past meals/diary entries keep working.
- `GET /api/foods?deleted=1` returns the **archive** side instead (the /audit Archived tab).
- `POST /api/foods/bulk` `{ ids, action: 'archive'|'restore'|'category', category? }` → `{ action, affected, ids }`. Idempotent (re-archiving affects 0). Not audited (audit_log revert only supports price_log).
- `POST /api/foods/from-nutrition` **un-archives** a food it reuses — deliberately re-adding a provider item shouldn't silently land on a hidden food.
- UI: `frontend/src/app/audit/page.tsx` (`/audit`, nav link) — Active/Archived tabs, search + category chips with counts, select-all-filtered, and a bulk bar (archive / restore / set category). Rows show the **flyer photo** (the fastest food-vs-TV signal); clicking the photo *or* the pencil opens the shared `FoodDetailModal` (there's deliberately no zoom lightbox here — the modal shows the photo alongside the names/prices/nutrition you'd want to fix). Both `stopPropagation()` so they don't toggle the row's checkbox. The thumbnails **must** stay `loading="lazy"` — this list is the whole catalog (hundreds of rows) and `GET /api/images/:id` serves full-resolution originals, so eager-loading them locks up the renderer.
- **Known gap:** re-running a scraper re-creates archived items (the worker fuzzy-matches against the *active* catalog, so an archived food looks like a new one). Archiving isn't yet a permanent "don't track this" signal.

**Merge duplicate foods (collapse N rows into one).** The catalog accretes duplicates (three "pork tenderloin" rows from different scrapes); `/audit` merges them.
- **`POST /api/foods/merge` `{ target_id, source_ids[] }`** → `{ target, merged }` (one transaction). Repoints **every** reference from the sources onto the survivor: `food_prices` / `food_macros` / `food_tags` (union onto target `ON CONFLICT DO NOTHING`, then drop source rows), `food_aliases` (each source's **name** + its aliases become aliases of the target, deduped via `idx_food_aliases_unique`), `meal_ingredients.food_id` + `consumption_logs.food_id` (UPDATE → target, so recipes/diary follow the survivor), and `price_logs.food_id` (origin owner the audit-log join reads). **`food_nutrition.food_id` is left as-is** — it's UNIQUE and the profile is already linked to the target via `food_macros`. Sources are **soft-deleted** (never hard-deleted); not audited (like `/api/foods/bulk`) — the archived sources are the trace, though their data has moved.
- **`POST /api/foods/merge-suggestions` `{ food_ids }`** → `{ groups: [{ food_ids, foods:[{id,name}], reason }], model }`. LLM duplicate-finder via `chatJson(system, user, MERGE_MODEL)` (falls back to `MEAL_MODEL`). Validated hard: unknown ids dropped, no id in two groups, groups of ≥2 only. **Proposes only** — never merges (same human-in-the-loop rule as OCR/auto-tag/meal). Capped at `MERGE_SUGGEST_MAX_FOODS` (60) per call; the UI batches the catalog. Env: `MERGE_MODEL` on `backend` (root `.env`, wired via docker-compose; `:free` per the OpenRouter-credits note).
- UI (`frontend/src/app/audit/page.tsx`): **`MergeModal`** (`modal.merge-foods`) — bulk-bar "Merge N" (≥2 selected, active tab); pick the survivor via radio, the rest archive. **`FindDuplicatesModal`** (`modal.find-duplicates`) — header "✨ Find duplicates" scans the active catalog in batches (mirrors `AutoTagModal`'s propose→review→apply + progress bar), each proposed group has a survivor radio and drop-food/dismiss-group controls, **Apply** loops `merge` per group. Both share the module-level `mergeFoods()` helper.

### Catalog tags & AI auto-tagger
Free-form labels on foods, managed from `/audit` (this is the "tag" half of the bulk clean-up story above).
- **Schema:** `tags` (name, unique **case-insensitively** via `idx_tags_name`) + `food_tags` (M:N; both FKs `ON DELETE CASCADE`). Every food read (`GET /api/foods` list **and** `GET /api/foods/:id`) carries `tags: [{id,name}]` via one shared subquery in server.ts, so tag chips render anywhere without an extra fetch. Merge unions `food_tags` onto the survivor (see Merge above).
- **Endpoints:** `GET /api/tags` (each with its count of **non-archived** foods), `POST /api/tags` (case-insensitive dedup — posting an existing name returns the existing row, no 409), `DELETE /api/tags/:id` (links cascade).
- **Two write shapes — don't confuse them:** bulk **same-tags** is `POST /api/foods/bulk` `{ ids, action: 'tag'|'untag', tag_ids }` (the same tag set applied/removed across every selected food); **per-food assignments** is `POST /api/foods/apply-tags` `{ assignments: [{ food_id, tag_ids }] }` (one transaction, `ON CONFLICT DO NOTHING`, returns `{ linked }`) — this is what a reviewed auto-tag draft commits.
- **AI auto-tagger:** `POST /api/foods/auto-tag` `{ food_ids, tag_ids, hint? }` asks a text-pool LLM (`chatJson`, `TAG_MODEL` override → falls back through the pool) which of the **chosen** tags fit each of the **chosen** foods. Capped at `AUTO_TAG_MAX_FOODS` (50) per call (free-model context limits — the UI batches). Validated hard: unknown food ids and invented tag names are dropped. Returns an **unsaved draft** reviewed in `AutoTagModal` (`modal.auto-tag`, propose→review→apply on `/audit`) and committed via `apply-tags` — the model never writes to the catalog (same human-in-the-loop rule as OCR/merge/meal drafting).

### Data model (`db/schema.sql`)
- `foods` (name, barcode, category, unit, `image_id` — user-chosen dashboard icon, nullable FK → `images`, see below, and `deleted_at` — **soft delete**, see Catalog audit above).
- `tags` + `food_tags`: free-form food labels, M:N — see Catalog tags above.
- `stores` (name, location, `latitude`/`longitude` learned from photo EXIF, `deleted_at` — **soft delete**). **Removing a store must stay a soft delete.** `price_logs.store_id` and `scrape_jobs.store_id` are `ON DELETE CASCADE`, so a real `DELETE FROM stores` would silently wipe that store's entire price history. **`GET /api/stores` filters `deleted_at IS NULL`** so removed stores vanish from every picker (review, dashboard, staging); reads that join `stores` **by id** (history, receipts, scan jobs) deliberately do **not** filter, so past prices still resolve the store's name — same convention as archived `foods`. There's no restore UI yet; clearing `deleted_at` in SQL is the recovery path.
- **Remove-a-store + reallocate** (one transaction): **`DELETE /api/stores/:id`** with body `{ reassign_to }` — a store id **moves every referencing row** (`price_logs`, `receipts`, `scan_jobs`, `scrape_jobs`) to that store; `null`/omitted **leaves them unassigned** (`store_id = NULL`). Then the store is soft-deleted. Returns `{ success, id, reassigned_to, moved: {…per-table counts} }`; 404 once gone, 400 on an unknown/self `reassign_to`. The denormalized `store_name` on `receipts`/`scrape_jobs` **follows the new store** when reassigning but is **kept as-is when unassigning**, so a receipt still records where the spend happened. **Not audited** — same precedent as `POST /api/foods/merge`, which likewise repoints `price_logs` in bulk. **`GET /api/stores/:id/usage`** → `{ price_logs, receipts, scan_jobs, scrape_jobs }` counts, shown in the confirm so the user sees what they're reallocating before committing.
- `price_logs`: `price`, `amount` + `amount_unit`, normalized `unit_price` (`DECIMAL(12,5)`), `is_sale`, **`sale_ends_at`** (DATE — see Sale expiry above), `source` (`scan|manual|scraper|queue`), and `deleted_at` (**soft delete**).
- `app_settings`: single row (id=1) of app-wide preferences — currently `default_sale_days` (see Sale expiry above).
- `audit_log`: before/after JSONB snapshot of every price_log **and consumption_log** mutation, with `reverted_at`. Revert (`POST /api/audit-log/:id/revert`) only supports `price_log` entries.
- `scan_jobs`: background OCR jobs. `status`: `staged` (uploaded, awaiting crop+send on `/staging`) → `queued` → `processing` → `done`/`failed` → `reviewed` (plus `discarded`); `done`/`failed`/`reviewed` can go **back** to `staged` via the re-stage endpoint above. Also `image_path`, `image_id`, `result` JSONB, `store_id`, and **`attempts` JSONB** (the per-model OCR trace — see the recovery section; NULL on jobs that predate it, which the UI handles).
- `images`: uploaded photos (`path`, EXIF `latitude`/`longitude`, and `original_image_id` — nullable self-FK marking a row as a crop of another image, set by the crop-before-OCR step above). Referenced by `foods.image_id`, `price_logs.image_id`, `scan_jobs.image_id`.
- `receipts`: one spending record per shopping trip (`store_id` **ON DELETE SET NULL** + denormalized `store_name` so spend survives a deleted store, `total`, `purchased_on` date, `item_count`, `source` `scan|manual`, `notes`, `image_id`, `scan_job_id`, `deleted_at` soft delete). See Budget / spending tracking below.
- `budget_goals`: single row (id=1, same bootstrap pattern as `user_goals`) holding the nullable `monthly_budget` target.

### Budget / spending tracking
- **Receipts are the spending record** — store source + total cost per trip. Written **automatically when a receipt scan is committed** in `ReviewItems` (one `POST /api/receipts` with `source='scan'`, linked to the source `image_id` + `scan_job_id`), and **by hand** on the budget page (`source='manual'`) for cash / un-scanned trips. Only receipt-type scans record one — price-tag and manual-item reviews don't.
- **`ReviewItems` receipt flow**: the scanner (`type==='receipt'` → `receiptCtx`) and inbox (`OpenReview.receipt`) pass a referentially-stable `receipt={ total, purchasedOn, scanJobId }` prop. The review shows an **editable receipt total + date** (prefilled from OCR `data.total`/`purchase_date`); on commit it records the row **once** (guarded by a `receiptSaved` flag) after ≥1 item saved. **The receipt's own total wins**; a blank total falls back to the summed saved item prices. The store is the review's selected store; item_count is how many items committed. No backfill for pre-existing commits — manual add covers history.
- **Endpoints** (`backend/src/server.ts`, near the end): `GET /api/receipts` (list; filters `?month=YYYY-MM`, `?store_id`, `?from`/`?to`), `GET /api/receipts/summary?month=` (this-month spend + `receipt_count`, `by_store`, last-12-months `by_month`, and the `monthly_budget`), `POST /api/receipts`, `PUT /api/receipts/:id` (partial; `store_id` is presence-based so it can be cleared), `DELETE /api/receipts/:id` (soft), and `GET`/`PUT /api/budget` (the single-row monthly target). Not audited (audit_log revert only supports price_log); soft delete is the recovery path.
- **UI**: `frontend/src/app/budget/page.tsx` (`/budget`, nav link in `layout.tsx`) — month nav, spent-vs-budget progress bar, by-store + by-month breakdowns (click a month bar to jump), receipts list with photo lightbox + edit/delete, and a manual add/edit popup (`ReceiptForm`, via the shared `Modal`, `modal.receipt-form`).

### Calorie tracking
- `food_nutrition`: per-serving label facts (`serving_size` + `serving_unit` in the shared unit vocabulary, `calories`, macros, and micronutrients). A profile is **shareable** — several foods can point at one via `food_macros` (see the M:N section) — so it's no longer strictly 1:1, though `food_nutrition.food_id` keeps a UNIQUE origin owner. **`PUT /api/foods/:id/nutrition` updates the food's currently-linked profile in place** (so an edit propagates to every food that shares it); only a food with no linked profile inserts+links a new one. The full nutrient column set is defined once as `NUTRIENT_FIELDS` in `backend/src/nutrition.ts` — server.ts builds its nutrition/consumption `INSERT`/`UPDATE`/`SUM` column lists from it, so **adding a nutrient = a schema migration + one entry in that array** (and matching display metadata in `frontend/src/lib/nutrition.ts` `MACRO_META`/`MICRO_META`). `GET /api/foods/:id` also returns `nutrition.shared_food_count` (how many foods link the effective profile).
- `consumption_logs`: diary entries. All nutrients (calories + macros + micros) are a **snapshot computed at log time** (`backend/src/nutrition.ts` `scaleNutrients` — supports the diary-only unit `serving`, deliberately not added to the synced `units.ts`); editing a food's facts never rewrites history. `food_name` is denormalized; soft delete + audited (`entity_type='consumption_log'`). `GET /api/diary` sums every nutrient column for the day's micronutrient totals.
- `user_goals`: single row (id=1) of daily calorie/macro targets. `GET /api/diary?date=YYYY-MM-DD` returns a day's entries + totals + goals. `consumed_at` is a naive timestamp — the client owns timezone handling (send local `consumed_at` and an explicit `?date=`).
- USDA FoodData Central lookup: `GET /api/nutrition-search?q=<text or barcode>` (`backend/src/fdc.ts`) proxies FDC search (key: `FDC_API_KEY` in root `.env`) and returns per-serving candidates; the nutrition editor in the dashboard food modal prefills from a picked candidate (`source: 'usda'`) and the user confirms before saving.
- Two USDA surfaces, both **user-initiated** (neither bulk-adds):
  - **Prefill** — `MacroEditor`'s built-in USDA search fills the facts form from a picked candidate (`source: 'usda'`); the user confirms + saves via `PUT /api/foods/:id/nutrition`. Edits an existing food; never creates rows on its own.
  - **Save-on-pick** — `frontend/src/components/NutritionSearch.tsx` (in the meals builder's "USDA database" tab) searches USDA and only saves the **one** result the user clicks "Add to meal" on, via `POST /api/foods/from-nutrition`, then drops it into the recipe. A recipe ingredient must reference a catalog food (FK), so the picked item becomes a real food — but a search no longer dumps every result.
- **LEGACY (kept, do not re-wire):** `NutritionSearch`'s **`autoSave`** mode auto-saved *every* search result — that piled up unwanted items, so it was removed from the dashboard ("Add from USDA"). The component's non-autoSave save-on-pick path is what the meals builder uses now; `from-nutrition`'s dedup-by-barcode/name + un-archive logic is documented in its route comment.
- **`frontend/src/components/FoodDetailModal.tsx`**: reusable food modal opened from a **diary** entry, the **inbox** review, the **/audit** table, and the **meals builder** (pencil on each ingredient row *and* on each catalog-picker suggestion, so a missing price/facts can be fixed inline — `onChange`/`onClose` refetch the catalog so the builder's live macro/cost preview updates). Lists a food's names, prices, and nutrition; add/remove names inline (`POST/DELETE /api/foods/:id/aliases`); view a price log's attached photo in a lightbox (`image_id` → `GET /api/images/:id`). **"Use another food's facts"** (`ShareNutritionModal`, `modal.share-nutrition`) searches foods that have nutrition and points this food at the picked one's profile (`POST /api/foods/:id/nutrition/copy-from/:sourceId`); a "shared with N foods" note appears when the effective profile is multi-linked. Price and macro editing are delegated to the two shared popups below — it only lists and launches them. Loads via `GET /api/foods/:id`.

### Per-100 nutrition basis — the nutrition twin of the canonical price display
A label's serving ("55 g", "1 cup") isn't comparable between foods, so **every surface that shows a food's facts also shows them per 100 g (solid) / 100 ml (liquid)**, exactly as prices are always shown per kg. The helpers live once in `frontend/src/lib/nutrition.ts` (display-only, **no backend twin** — like `canonicalUnitPrice` in `units.ts`): `per100Basis` (picks the basis), `nutrientsPer100` (facts scaled to 100), `formatCaloriesPer100` (`"455 kcal / 100 g"`). Don't re-derive this inline — call the helper.
- **Which basis:** the dimension of `serving_unit`, falling back to the food's own `unit`; `volume` → 100 ml, `mass` → 100 g. Anything else (a serving counted in `each`) returns **null** — 100 g of "1 each" is unknowable without a per-item weight, which isn't modelled. **Every caller must handle that null** by falling back to the per-serving figure with its basis spelled out, never by printing a bare calorie count.
- Surfaces: the dashboard's `modal.price-history` and the shared `FoodDetailModal` show a per-serving chip **plus** a per-100 chip (both, so the label as printed is still visible); the dashboard **kcal /100** column and the `/audit` Nutrition column read per-100 outright.

### Meal plans (recipes)
- `meals` (name, notes, `servings` = portions the recipe makes) + `meal_ingredients` (meal_id, food_id, `amount` + `amount_unit` — shared unit vocab plus the diary-only `serving`). **Nothing is stored for macros/cost** — every read (`GET /api/meals`, `GET /api/meals/:id`) computes them live via `backend/src/meals.ts` `summarizeMeal`: per-ingredient `scaleNutrients` + `ingredientCost`, meal totals, and per-serving figures (totals ÷ servings), with `nutrition_complete` / `cost_complete` flags when an ingredient lacks facts or a priced purchase.
- **Cost rule** (`ingredientCost`): the food's latest non-deleted price *with a non-null `unit_price`* (through the `food_prices` join) × the ingredient amount in base units. Mass↔volume mismatches convert through `foods.density` (same rule as the canonical display); `serving` converts via the food's `serving_size`/`serving_unit`; count↔mass/volume is unresolvable → cost `null` for that row. Raw price only — `usable_pct` is deliberately **not** applied.
- Endpoints: `GET/POST /api/meals`, `GET/PUT/DELETE /api/meals/:id` (PUT with `ingredients` replaces the list wholesale), `POST /api/meals/:id/clone` ("Copy of X"), `POST /api/meals/:id/log`. Meals are **hard-deleted** (not audited; clone covers recovery). `validateIngredient` (meals.ts) 400s unknown units.
- **Logging to the diary creates ONE consumption_log entry** (not one per ingredient): per-serving nutrients × portions, `food_id = NULL`, `food_name` = meal name, `amount_unit = 'serving'`, `source = 'meal'`, and a nullable `consumption_logs.meal_id` for provenance (`ON DELETE SET NULL` — diary snapshots outlive the meal). It goes through the same transaction/audit path as manual entries via the shared `insertConsumptionLog` helper in server.ts — any new diary-writing endpoint should use that helper.
- **AI meal drafting**: `POST /api/meals/generate` (`{ food_ids, targets?, notes? }`) prompts an OpenRouter chat model via `chatJson` (`backend/src/llm.ts`, a thin shim over the **text model pool** in `backend/src/modelPool.ts` — `response_format: json_object`, sequential failover across `FREE_TEXT_MODELS` (+ paid when `USE_PAID_MODELS`) because free models return empty/malformed content) with the selected foods' per-serving facts; targets are **per serving of the meal**. The response is validated (unknown food_ids/units dropped) and returned as an **unsaved draft** the UI loads into the builder — the user reviews and saves explicitly, same review invariant as OCR. Env on `backend`: `OPENROUTER_API_KEY` + the `FREE_TEXT_MODELS`/`PAID_TEXT_MODELS` pool (root `.env`; empty `FREE_TEXT_MODELS` seeds from legacy `MEAL_MODEL`; the account has no paid OpenRouter credits, so keep the free list to `:free` models — currently `nvidia/nemotron-nano-12b-v2-vl:free`).
- UI: `frontend/src/app/meals/page.tsx` (nav link in `layout.tsx`). The builder's live totals/cost preview mirrors the backend rules client-side (display-only; `previewCost` in the page must stay consistent with `backend/src/meals.ts` `ingredientCost`); the server's figures are authoritative. The ingredient picker has two tabs — **Catalog** (fuzzy search over existing foods) and **USDA database** (the shared `<NutritionSearch>` in **save-on-pick** mode: results are NOT saved as they load; clicking "Add to meal" saves just that one via `from-nutrition` and drops it in at its USDA serving weight). A food with nutrition defaults its ingredient amount to its serving weight (e.g. `28 g`) via the shared `rowDefaults` helper (used by both add paths). Future meal-feature ideas live in [ROADMAP.md](ROADMAP.md).

### Foods ↔ prices/nutrition are many-to-many
- Join tables `food_prices(food_id, price_log_id)` and `food_macros(food_id, nutrition_id)` are the authoritative food→x links for reads; `price_logs.food_id` / `food_nutrition.food_id` remain as the **origin owner** (audit + back-compat) and are kept in sync on create. A food's list/detail prices and its effective nutrition (owned preferred, else first linked) are read **through the joins**. Backfilled from the origin columns; the audit-log join still uses `pl.food_id`.
- Link/unlink an existing price or nutrition profile to additional foods: `POST/DELETE /api/foods/:id/prices/:priceLogId/link` and `POST/DELETE /api/foods/:id/macros/:nutritionId/link`.
- `food_nutrition.food_id` keeps a UNIQUE (one *owned* profile per food); sharing across foods happens via `food_macros`. **Two different products can share one nutrition profile** (the "same underlying facts" case): `POST /api/foods/:id/nutrition/copy-from/:sourceId` unlinks this food's current profile and links the source food's, garbage-collecting a now-orphaned owned profile. From then on `PUT .../nutrition` on **either** food edits the one shared profile (it updates the linked profile by id, never re-forks). `DELETE /api/foods/:id/nutrition` only drops a profile no other food still links (a shared one is just unlinked). UI: `FoodDetailModal`'s "Use another food's facts".

### Shared image cropper — reuse for any crop UI, don't re-implement
`frontend/src/components/ImageCropper.tsx` is the single crop surface (react-easy-crop pan/zoom + a zoom slider + Back / Use-original / primary buttons). It is **presentation-only** — embed it inside a `Modal` (it does not wrap one), so it works both in `FoodIconPicker`'s modal and the scanner's `modal.scan-crop`. It emits a cropped JPEG `Blob` via `onCropped`; the canvas work lives in `frontend/src/lib/crop.ts` `cropImageToBlob(url, area, { maxSize?, quality? })` (aspect-preserving — omit `aspect` to frame the whole image and derive the shape from it, pass `1` for a square). Used by `FoodIconPicker` (square, ≤512px) and the scanner's crop-before-OCR modal (≤2048px). Any new crop UI reuses this — don't re-implement a canvas `getCroppedBlob`. `data-loc="component.image-cropper"`.

### Shared class vocabulary (`globals.css` `@layer components`) — reuse, don't re-paste
The recurring Tailwind strings are named once in `frontend/src/app/globals.css` inside `@layer components`: **`.card`** (the glass surface = `glass-panel` look + `rounded-2xl`; padding/accent/other-radius stay inline), **`.panel`** (the semi-solid `bg-slate-950/60 border border-white/5 rounded-xl` *inner* surface nested inside cards — efficiency rows, diary tiles, meal ingredient rows), **`.field-label`**, **`.field-input`** (text input / `<select>`), **`.btn`** + **`.btn-primary`** (violet→indigo gradient) / **`.btn-secondary`** (white/5 ghost), and **`.badge`** (the small pill/tag shape — `font-bold uppercase px-2 py-0.5 rounded-full border` — used for category chips, status/source tags, and approval states across the dashboard, `FoodDetailModal`, `ReviewItems`, inbox, scrapes, diary, and history pages). Reuse these instead of re-pasting the raw utilities; keep the parts that genuinely vary per site (padding, width, accent color, a different radius, text size) as inline utilities — e.g. `.badge` never bakes in color or `text-[9px]`/`text-[10px]` sizing, and a non-uppercase badge (diary's per-meal kcal chip) adds `normal-case` inline to override it.
- **They must live in `@layer components`, not as plain rules.** The components layer is emitted *before* Tailwind's utilities layer, so an inline utility still overrides the same property in one of these classes — that's why `card rounded-3xl` renders at 24px (the hero) and `field-input focus:border-emerald-500` re-tints the focus ring. A plain (unlayered) rule defined after `@tailwind utilities` (like the standalone `.glass-panel`/`.glass-panel-hover`, which stay as-is for the hover-lift cards) would instead *win* over inline utilities — that's exactly why the old `glass-panel border border-white/5` rendered a 0.06 white border, not the `border-white/5` you'd expect, and why `.card` bakes the 0.06 border in as raw CSS to stay pixel-identical.
- `@apply` can only reference Tailwind utilities, **not** the plain `.glass-panel` class — so `.card` inlines the glass background/blur as raw CSS rather than `@apply glass-panel`.
- Only genuinely-repeated strings are named; one-off/compact controls (the dashboard store & add-food forms, emerald/teal accent buttons) stay inline on purpose — don't mint a class per variant.
- `.btn` bundles a uniform `active:scale-[0.98]`/`disabled:opacity-50`; buttons converted to it gain that press/disabled feel (visually identical at rest — an intentional consistency win, not a regression).

### Section-comment labels + `data-loc` (inspect-element → source)
Major JSX regions carry a scannable banner comment — `{/* ═══ Section: <name> ═══ */}` — so a region can be named directly ("edit the Summary Cards section"). Fine-grained sub-comments stay as plain `{/* ... */}`. **These comments are source-only** — React/SWC strip JSX comments at compile time, so they never reach the browser DOM and inspect-element can't show them.

For that, every major region also carries a **rendered** `data-loc="<area>.<region>"` attribute (kebab-case dotted slug, e.g. `data-loc="dashboard.summary-cards"`, `data-loc="modal.price-editor"`) placed directly on the element the `Section:` comment precedes, so a grep of the slug finds both the source landmark and the live DOM node. Nothing in `next.config.js`/Tailwind strips `data-*` attributes, so it survives in both build targets (Docker `next build` and the `BUILD_TARGET=static` Capacitor export). **To point Claude at a region: inspect the element in the browser, read the nearest `data-loc`, and say "edit data-loc `<value>`"** — Claude greps for `<value>` (matching either `data-loc="<value>"` on a plain element or `dataLoc="<value>"` where it's passed as a prop into `Modal`, see below) and lands on the exact JSX. Page roots use `page.<name>`, layout landmarks (`app/layout.tsx`) use `layout.header`/`layout.main`/`layout.footer`, shared components use `component.<name>`.

Portaled modals are the one gotcha: everything routed through the shared `Modal` (see below) renders via `createPortal` into `document.body`, detached from the page-root subtree, so they're tagged independently via `Modal`'s `dataLoc` prop (e.g. `dataLoc="modal.food-detail"` in the *caller's* JSX) rather than by adding `data-loc` inline — the prop is spelled `dataLoc` (camelCase) in source but renders as `data-loc` (kebab-case) in the DOM, so grep the slug value itself, not a fixed `data-loc=`/`dataLoc=` prefix. When you add a new `Section:` banner, add the matching `data-loc` alongside it — the two travel together.

### Two shared popups for entering price / macros — reuse these, don't fork
There are exactly **two** input surfaces for a food's price and nutrition, both **popup modals** (via the shared `Modal`), reused everywhere:
- `frontend/src/components/PriceEditor.tsx` — one price log. **Add** (no `log` → `POST /api/foods/:foodId/prices`) or **edit** (`log` → audited `PUT /api/price-logs/:id`, optional `DELETE`). Props: `{ foodId, foodName?, log?, stores?, onClose, onSaved, onDeleted? }`.
- `frontend/src/components/MacroEditor.tsx` — one food's per-serving nutrition facts (serving, calories, macros, collapsible micros) + USDA search. `PUT /api/foods/:id/nutrition`. Props: `{ foodId, foodName?, barcode?, nutrition?, onClose, onSaved }`.

Both are launched (never re-implemented) from: the **dashboard** food modal, the **diary**/**inbox** `FoodDetailModal`, and — PriceEditor also — the **history** table (Edit → popup). If you need a price/macro form anywhere, open one of these. (`ReviewItems`' bulk scan grid is a separate concern: it edits *extracted* items pre-commit, not existing price_logs/foods.) Known names are add/rename/delete-able from the dashboard food modal and add/delete-able from `FoodDetailModal`.

### Sale expiry (`price_logs.sale_ends_at`) — an expired sale is not a current price
A sale price stops being real the day the sale ends, so it must stop being quoted.
- **`price_logs.sale_ends_at` (DATE, nullable)** is the last day a sale price is valid. Only meaningful with `is_sale = true`; a non-sale price never expires, so **`resolveSaleEndsAt` drops any date passed with `is_sale = false`** and the pair can never contradict each other. That's also why `sale_ends_at` is **not** in `EDITABLE_PRICE_FIELDS` — it's derived in the handler, not merged blindly.
- **`ACTIVE_PRICE_SQL`** (`backend/src/server.ts`, defined once near the top) is the predicate `(pl.is_sale = false OR pl.sale_ends_at IS NULL OR pl.sale_ends_at >= CURRENT_DATE)`. **Every read that surfaces a food's CURRENT price must apply it on top of `deleted_at IS NULL`** — currently `GET /api/foods`'s `latest_prices` subquery, `GET /api/prices/efficiency`, and `MEAL_INGREDIENT_SELECT`'s `latest_price` (so meal costs don't quote a dead sale). It expects the table aliased `pl`. Adding a fourth current-price read means adding this predicate too; a drift means the dashboard quotes a price the store stopped honouring.
- **Reads that show the PAST deliberately do NOT filter it** — `GET /api/foods/:id/prices` and `GET /api/price-logs` (history) still list expired sales, because the sale genuinely happened. Same convention as archived foods resolving by id.
- **Where the date comes from**, in priority order: (1) an explicit date from the caller — the scan read one, or the user set it in the inbox/PriceEditor; (2) the flyer's own end date for scraped prices (`item.valid_to` for Flipp, `item.expires_on` for cocowest — both were already fetched and discarded before); (3) **`app_settings.default_sale_days`**, today + N. A sale must end up with *some* date, or a discounted price would be quoted as current forever.
- **`app_settings`** is a single row (id = 1, same bootstrap pattern as `user_goals`/`budget_goals`) behind **`GET`/`PUT /api/settings`** (`{ default_sale_days }`, validated 1–365). UI: `frontend/src/app/settings/page.tsx` (`/settings`, nav link) — presets + custom, with a live "would expire on …" preview.
- **OCR extracts it**: `sale_ends_at` (ISO `YYYY-MM-DD`) on both `ReceiptItem` (alongside a new `is_sale`) and `PriceTagData` — the usual hand-synced pair, `ocr-service/app/models.py` ↔ `frontend/src/types/scan.ts`. `prompts.py` tells the model to read the END of a date range and, importantly, **not** to confuse a sale end date with a best-before date, the purchase date, or the receipt timestamp, and to return null rather than guess a duration (guessing is the app default's job, not the model's).
- **Inbox override**: `ReviewItems`' grid has a **Sale ends** column — a per-row sale checkbox plus a date input that shows the *effective* date (`effectiveSaleEnd` = the row's own date, else the app default rendered in italics). A batch control (`review-items.sale-expiry`) sets every sale row at once, since one scan is usually one shop or one flyer. Commit sends the **displayed** date explicitly, so what the user saw is what gets stored rather than the backend re-deriving a default that could disagree.

### Price-per-unit normalization
`amount` + `amount_unit` (e.g. `2 lb`) normalize to a base unit (grams / ml / each) to produce `unit_price` (price per base unit) for cross-package comparison. The backend (`units.ts`, used in `POST/PUT /api/foods/:id/prices`) is the source of truth; the frontend copy is display-only.

**Canonical dashboard display (kg standard).** Every price shown on the dashboard table / PriceEditor preview / FoodDetailModal price lines goes through `formatCanonicalUnitPrice` — or, where a *number* is needed rather than a string (the dashboard table's sortable Best-price column), its numeric twin `canonicalUnitPrice` that the formatter wraps (`frontend/src/lib/units.ts`, display-only, no backend twin). It reads the price in **one standard unit per dimension** so rows are comparable regardless of how they were entered: **mass → per kg**, **volume → per kg** (via `foods.density`, see below), **count → per each**. When the as-entered unit isn't already the canonical one it's appended in brackets **as purchased** — total price over the purchased amount, not reduced to a per-single-unit figure (so a 600 g pack reads `$7.48/kg ($4.49/600g)`, never `$0.01/g`). The purchased amount is omitted when it's 1: `$4.41/kg ($2.00/lb)`, `$1.20/kg ($1.20/L)`, `$0.25/each ($3.00/dozen)`. This required the `GET /api/foods` `latest_prices` subquery to also select `pl.amount, pl.amount_unit` (the client computes the per-kg value from price+amount+unit, not from `unit_price`). Count can't be expressed per kg without a per-item weight (not modelled), so count stays per-each.

### Density (`foods.density`)
`foods.density` (`DECIMAL(8,4)`, default 1, must be > 0) is kilograms per litre, used **only** to convert a per-volume price into a per-kg price for the canonical display above (water ≈ 1, oil ≈ 0.92, honey ≈ 1.42). Display/value-only: it does **not** touch the stored `unit_price` (still per ml) nor nutrition. Accepted on `POST`/`PUT /api/foods` (validated > 0), returned via `SELECT f.*`. Edited from the dashboard food modal and `FoodDetailModal`, where the density input is shown **only when the food's `unit` is a volume unit** (`normalizeUnit(unit)?.dimension === 'volume'`); passed as `density` into `PriceEditor` for its live preview.

### Usable portion (`foods.usable_pct`)
`foods.usable_pct` (`DECIMAL(6,2)`, default 100, must be > 0) is the percent of a purchased amount that is actually usable — e.g. 70 for chicken drumsticks (30% bone), or **>100** for dry goods that expand when prepared (dry lentils → cooked). Edited via `PUT /api/foods/:id` (accepted on `POST` too); returned on every `GET /api/foods*` via `SELECT f.*`. It is **display/value-only**: `frontend/src/lib/units.ts` `formatUnitPrice(price, amount, unit, usablePct?)` divides the per-unit price by `usablePct/100` and suffixes " usable" (an effective cost per *usable* unit). It deliberately does **not** touch nutrition scaling (labels are already per edible serving) nor the stored `unit_price` (raw, audited, and shareable across foods via the M:N join). The efficiency endpoint compares prices within one food, so `usable_pct` cancels there — no change. Editable from the dashboard food modal and `FoodDetailModal`; passed as `usablePct` into `PriceEditor` for its live preview.

### Dashboard food list: a client-side sortable table
The dashboard (`frontend/src/app/page.tsx`) renders the catalog as a **sortable/filterable table** (`data-loc="dashboard.inventory-table"`), not the old card grid. It **loads the whole catalog once per page load** (`GET /api/foods` with **no** `limit` → plain array) and does search, filtering, sorting and paging **entirely in memory**; `fetchFoods()` is re-run only after a mutation (add food, price edit, icon change) to refresh that cache.
- **Why client-side:** measured at ~575 B/food (325 foods ≈ 182 KiB, ~50 ms; it gzips ~8x). A few thousand foods is still a few hundred KB — cheap next to a round-trip per keystroke. Crucially it's also the only way column sorting can be *honest*: with server-side paging, sorting would only reorder the 24 rows already on screen. **Don't reintroduce server paging here without moving sorting into SQL too.**
- Sort state is `sortKey`/`sortDir` via the `SortHeader` component; text columns start ascending, numeric ones descending. **Missing values always sort last in both directions**, so flipping a column never fills the top with blanks. Sortable: name, category, best price, store count, kcal, last-price date.
- The **kcal /100** column reads **per 100 g / 100 ml** (see Per-100 nutrition basis below), not per label serving, so the numbers are comparable — same rationale as the canonical Best price column. It sorts on that per-100 value; a food whose serving unit can't be converted (counted in `each`) falls back to its per-serving figure and labels the basis inline.
- The **Best price** column sorts on `canonicalUnitPrice` (`frontend/src/lib/units.ts`) — the numeric twin of `formatCanonicalUnitPrice`, which is now a thin display wrapper around it so the sorted number and the shown string can never disagree. It's the *cheapest* canonical price across the food's latest prices (mass/volume → per kg, count → per each).
- Filters: search (name/barcode/description/**aliases**), category chips (derived from the loaded catalog, no server `categories` needed), plus **Has price** / **On sale** toggles. Page size is user-selectable (25/50/100/Show all).
- **`GET /api/foods`'s `limit`/`offset` branch (`{ foods, total, categories }`) is still supported and smoke-tested, but no UI calls it any more.** The full-array form is what every caller uses (dashboard, `ReviewItems` fuzzy-match, meals/diary pickers) — **do not add `limit` to those**.

Every food row (both list and `GET /api/foods/:id`) also carries `display_image_id = COALESCE(foods.image_id, earliest non-deleted image_id from the food's linked price logs)` — the dashboard row thumbnail (kept `loading="lazy"`, same reason as `/audit`). `foods.image_id` (nullable FK → `images`) is the user's explicit icon choice, set via the popup opened by clicking a row's thumbnail (the click `stopPropagation()`s so it doesn't open the row's detail modal) (`frontend/src/components/FoodIconPicker.tsx`): pick one of the food's saved scan/scrape images (fetched via the existing `GET /api/foods/:id/prices`, deduped by `image_id`) or upload a new one, then crop to a square via the shared **`<ImageCropper>`** (canvas crop in `frontend/src/lib/crop.ts` `cropImageToBlob`, ≤512×512 JPEG here) before `POST /api/images` + `PUT /api/foods/:id { image_id }`. `PUT /api/foods/:id` is otherwise pure `COALESCE` (omitted field = unchanged), but `image_id` is **presence-based** since COALESCE can never express "set to null": key absent → untouched, `{image_id: null}` → clears back to the `display_image_id` fallback (the picker's "Reset to default"), `{image_id: <n>}` → set (400 if it doesn't reference a saved `images` row). Re-cropping never deletes the previous `images` row (images are shared/immutable elsewhere too — scrapes, scans); orphaned icon images are an accepted tradeoff, not a leak to fix. The **`modal.price-history` detail modal shows the same `display_image_id` photo** in its header (a larger thumbnail beside the name), and clicking it opens that same `FoodIconPicker` — it stacks above the detail modal (`z-[60]` over `z-50`) and, on save, refreshes the open modal via `refreshSelectedFood` (not just `fetchFoods`) so the new photo appears in place.

### History, audit & revert
Every price_log create/update/delete flows through `backend/src/audit.ts` `recordAudit` **inside the same transaction** as the mutation. Deletes are soft (`deleted_at`); revert (`POST /api/audit-log/:id/revert`) applies the inverse and is itself audited (so reverts are revertible). Any query that surfaces a "current" price — `GET /api/foods` `latest_prices` subquery and `GET /api/prices/efficiency` — must filter `deleted_at IS NULL`. UI: `frontend/src/app/history/page.tsx`.

### Scraper (Flipp flyers)
`worker/src/worker.ts` also processes `scraping-queue` (triggered by `POST /api/scrape/:storeId`, body `{ postal_code?, query? }`; postal code falls back to `FLIPP_POSTAL_CODE` in root `.env`). `worker/src/flipp.ts` hits Flipp's public flyer JSON API (`backflipp.wishabi.com/flipp/items/search` — the flipp.com backend; **no Playwright/browser**, the worker image is plain `node:20-slim`) and keeps items whose merchant fuzzy-matches the store name — so the store must be named like the Flipp merchant ("Walmart", "Save-On-Foods"). With a `query` it logs up to 15 matching deals (creating foods with category `Scraped` when nothing in the catalog matches); without one it searches each catalog food (capped at 50, rate-limited) and logs the single best-scoring current deal per food. Pack sizes are parsed from flyer item names/affixes ("473mL", "2 x 96g", "/lb", "2/" multi-buys) into `amount` + `amount_unit`. The worker writes **through the backend REST API** (`BACKEND_URL`, i.e. `POST /api/foods/:id/prices` with `source: 'scraper'`) — never direct SQL — so scraped prices get unit normalization, their `food_prices` join row (without it they'd be invisible to reads), and an audit entry like every other source.

**Progress tracking + source images.** `POST /api/scrape/:storeId` first inserts a `scrape_jobs` row (`db/schema.sql`) and passes its id on the queue job as `scrapeJobId`; the response includes `scrapeJobId`. The worker updates that row's `status`/`phase`/`total`/`processed`/`logged` as it runs and **appends one detail record per logged price** to `items` JSONB (food, flyer name, price, `image_id`, `flyer_url`). These progress writes go **direct via the worker's pg pool** (bookkeeping, like the OCR path's `scan_jobs` updates) — only the *prices* go through the audited backend API. For each logged price the worker downloads the item's Flipp **clipping image** (`clipping_image_url`) and registers it as an `images` row by POSTing multipart to `POST /api/images` (reusing the photo path), then attaches the returned `image_id` to the price log — so scraped prices show their real flyer artwork in the existing `image_id` lightbox everywhere (FoodDetailModal, history). `flipp.ts` `flippItemUrl` builds `https://flipp.com/en-ca/item/<flyer_item_id>` — the deep link to the flyer page the item came from (**Flipp's JSON API exposes only per-item clipping images, no full-page rasters**, so "the relevant page" is a link, not a saved image). Reads: `GET /api/scrape-jobs` (list, `item_count` only) and `GET /api/scrape-jobs/:id` (full `items`). UI: `frontend/src/app/scrapes/page.tsx` — live progress list (polls every 2s while any job is `queued`/`processing`), expandable per-run to show each logged price with its saved flyer thumbnail (lightbox) and a "Flyer page ↗" link; the dashboard scrape form redirects to `/scrapes?job=<id>` (auto-expands, needs a `<Suspense>` boundary for `useSearchParams`).

### Second scraper source: cocowest.ca Costco sale posts
`worker/src/cocowest.ts` scrapes cocowest.ca "weekend update" blog posts, which list Costco BC/AB/SK/MB flyer sale items. Triggered by `POST /api/scrape-cocowest`, body `{ store_id, url }` (`url` must match `cocowest.ca`) — no postal code or merchant matching, since the whole post belongs to whichever store the caller selects (create a "Costco" store first). Shares the same `scrape_jobs` table and `scraping-queue` as Flipp: `backend/src/queue.ts` `addCocowestScrapeJob` enqueues `{ ..., source: 'cocowest', url }`, and `worker/src/worker.ts`'s queue handler branches on `job.data.source` (`'flipp'` is the default for back-compat with jobs already in flight) to call `scrapeCocowest` instead of `scrapeFlippFlyers`. `scrapeCocowest` mirrors Flipp's *query mode*: log a price for **every** parseable item, creating a food with **category `Costco`** when nothing in the catalog fuzzy-matches (the confirmed scope choice — the post mixes grocery with electronics/appliances/toys, and everything gets logged rather than filtered).

- **No JSON API, no DOM parser.** cocowest is a WordPress blog; each item is one `<img>` whose `alt` text carries `ITEM# NAME SIZE ($SAVINGS INSTANT SAVINGS EXPIRES ON YYYY-MM-DD)? $PRICE` and whose `data-src` (not `src`, which is an inline SVG lazyload placeholder) is the real product photo. `cocowest.ts` parses this with regex, not an HTML/DOM parser — it keeps the worker's plain `node:20-slim` image dependency-free, and is actually more robust here: some product names contain an unescaped `"` (e.g. `ACER 27"...`), which truncates the `alt` attribute in the raw HTML itself (not a parser bug). Those items have no trailing `$price` after truncation and are silently skipped — in practice this is ~40 of ~380 images per post, almost all non-grocery electronics/TVs.
- **Shared parsing/matching.** Amount/unit parsing (`parseAmount`) and catalog fuzzy-matching (`nameMatchScore`, `symmetricMatchScore`) were pulled out of `flipp.ts` into `worker/src/scrape-common.ts` so both scrapers use one implementation; `flipp.ts`'s `parseFlippAmount` is now a thin wrapper that reads the multi-buy quantity from Flipp's `pre_price_text` and delegates. `scrape-common.ts` also adds a `PACK OF (\d+)` pattern (cocowest phrasing) alongside Flipp's `N PK`/`N CT` forms.
- **Images and progress.** Per logged item, `saveFlyerImage(item.image_url)` downloads the cocowest product photo and registers it via `POST /api/images` exactly like the Flipp path, so cocowest-sourced prices get the same `image_id` lightbox everywhere. `flyer_url` in each `scrape_jobs.items` record is the cocowest **post URL** (not a per-item deep link — cocowest has no per-item page).
- **Schema:** `scrape_jobs.source` (`'flipp'|'cocowest'`, default `'flipp'`) and `scrape_jobs.source_url` (cocowest's post URL) — added via the usual `ADD COLUMN IF NOT EXISTS` pattern in `db/schema.sql`, so existing DBs need the manual `ALTER TABLE` too (see the schema.sql gotcha above).
